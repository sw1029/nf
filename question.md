사용자가 제공한 `plan/user_request.md` (요구사항 원본)과 현재 구현된 코드(`engine.py`, `rule_extractor.py`, `user_ui.html`), 그리고 현황 보고서(`IMPLEMENTATION_STATUS.md`)를 종합적으로 분석한 결과입니다.

현재 구현은 **핵심 아키텍처(오케스트레이터-워커 분리, FTS/Vector 이중 검색, 정합성 검토 엔진)**는 요구사항을 충실히 따르고 있으나, **세부적인 사용자 경험(UX) 및 일부 고난도 기능(시맨틱 타임라인, 능동적 태깅 인터랙션)**에서 미비점이 발견됩니다.

### 1. 정합성 검토 (Consistency Check) 분석

**요구사항 대비 구현 현황:**

* **[충족] 명시적 필드 중심 검사 (Requirement 3):**
* `rule_extractor.py`에 나이, 시간, 장소, 관계 등을 정규식으로 추출하는 로직이 구현되어 있습니다.
* `engine.py`의 `_compare_slot` 함수에서 정수(int), 불리언(death), 문자열 유사도 등을 기반으로 '사실/시간' 등의 하드 제약을 우선 검사하는 로직이 존재합니다.


* **[충족] 신뢰성 점수 분해 (Requirement 3):**
* `ReliabilityBreakdown`을 통해 FTS 강도, 근거 개수, 확정 근거, 모델 점수를 분해하여 제공하고 있습니다.


* **[충족] Unknown 및 예외 처리 (Requirement 3, 12):**
* `Verdict.UNKNOWN` 상태를 정의하고, 근거가 상충하거나 모호할 경우(Unknown Reasons)를 명시합니다.
* 화이트리스트(`whitelist_repo`)와 무시(`ignore_repo`) 기능이 백엔드와 UI에 구현되어 있습니다.



**미비점 및 개선점:**

* **1. '서술 트릭/거짓말'의 의미론적 구분 부재:**
* **현황:** 현재는 단순 `VIOLATE`(충돌) -> 사용자 수동 `Whitelist`(예외) 처리로만 구현되어 있습니다.
* **요구사항:** "의도된 모순/거짓말은... 사용자가 체크 시 화이트리스트 처리 + **추가 정보 형식으로 저장**"
* **개선안:** UI에서 예외 처리 시 단순 '허용' 뿐만 아니라 **"의도된 거짓말(Lie)"** 혹은 **"서술 트릭"**이라는 메타태그를 달 수 있게 하고, 이를 저장소에 별도 속성으로 남겨 추후 RAG나 분석 시 해당 문장이 '거짓'임을 AI가 인지하도록 데이터 구조를 확장해야 합니다.


* **2. 문장 분절(Segmentation)의 정밀도 이슈:**
* **현황:** `_segment_text` 함수가 마침표(.) 등 단순 문장 부호 기준으로 작동합니다.
* **요구사항:** "High precision, Low recall" (확실한 것만 잡기)
* **개선안:** 소설은 대화문("...")이나 문장 부호의 변칙적 사용이 많습니다. 단순 Rule Base 분절은 오탐(False Positive) 가능성이 큽니다. 문맥을 고려하는 **Semantic Chunking** 혹은 **spacy/kss** 같은 NLP 라이브러리를 도입하여 분절 정확도를 높여야 UX 방해를 줄일 수 있습니다.


* **3. 성능 병목 (Critical):**
* `IMPLEMENTATION_STATUS.md`에 따르면 정합성 검사가 800화 기준 수 초~수십 초가 걸립니다. 이는 실시간 검토 UX를 해칩니다.
* **개선안:** 현재 '상시 검토'가 백그라운드에서 돌지만, UI상에서 사용자가 입력하는 즉시 반응하기 어렵습니다. **Diff 기반의 부분 검사(Incremental Check)** 로직을 강화하여 변경된 문단만 0.5초 이내에 검사하도록 최적화가 시급합니다.

### 2. 개선 제안 (Improvement Suggestion) 분석

**요구사항 대비 구현 현황:**

* **[부분 충족] 제안 및 근거 인용 (Requirement 6):**
* UI에 '문장 교정(Propose)' 탭이 존재하며, 백엔드에 RAG 기반 제안 구조가 잡혀 있습니다.


* **[미비] 구조화된 스키마 기반 제안:**
* **요구사항:** "사용자 작성 문서를 별도로 schema 형식으로 변환하는 과정... 문서ID/섹션/태그 경로까지 보여주는 방식"
* **현황:** `engine.py`는 정합성 체크에 집중되어 있고, 개선 제안 시 사용자의 입력문을 구조화된 데이터(Schema)로 변환하여 "이 캐릭터는 설정상 존댓말을 써야 함"과 같이 **제약 기반 교정**을 수행하는 구체적인 로직(Controller 레벨)이 코드상 명확지 않습니다.
* **개선안:** 단순 LLM의 문장 윤문(Paraphrasing)을 넘어, **[설정 검사 -> 위배 사항 감지 -> 이를 반영한 프롬프트 생성 -> 문장 재생성]**의 파이프라인을 명시적으로 구축해야 합니다.

### 3. UI/UX 완성도 및 개선 방안 (`user_ui.html` 분석)

**현재 UI의 장점:**

* **Clean Layout:** 좌측 탐색기, 중앙 에디터, 우측 보조 도구(Assistant)의 3단 구성은 데스크톱 환경(Windows)에 최적화된 전형적이고 효율적인 IDE 레이아웃입니다.
* **Non-blocking UX:** 백그라운드 작업(Jobs) 상태를 배지(Badge)와 팝업 패널로 보여주어 사용자가 작업을 멈추지 않게 한 점이 훌륭합니다.
* **Visual Feedback:** 에디터 내에 `nf-violate-bg` 등으로 밑줄을 긋고, 클릭 시 카드와 연결되는 상호작용은 직관적입니다.

**UI/UX 미비점 및 개선 방안:**

* **1. 타임라인(Timeline)의 개념 불일치 (Requirement 8-2):**
* **현황:** 현재 'TIMELINE' 탭은 파일/챕터 구조를 보여주는 **Git Graph 스타일의 구조도**(`renderTimeline`)로 구현되어 있습니다.
* **요구사항:** "세계관 타임라인 문서... 상대 서술 시점... timeline_idx"
* **개선안:** 파일 구조도가 아닌, **실제 소설 속 시간(Year/Date)** 을 축으로 하는 **간트 차트(Gantt Chart)** 혹은 **수직 타임라인 UI**가 필요합니다. 각 에피소드가 세계관의 어느 시점에 위치하는지 매핑하는 별도의 메타데이터 입력 UI가 타임라인 탭에 추가되어야 합니다.


* **2. 사용자 참여 기반 태깅 부재 (Requirement 9):**
* **요구사항:** "드래그 후 해당 문장에 태깅(복선과 같은 태그)와 같이 사용자 참여 기반 지정"
* **현황:** 에디터(`user_ui.html`)에는 텍스트 작성과 스타일링(굵게, 기울임 등) 기능만 있고, **특정 문장을 드래그하여 '이것은 복선임', '이것은 중요 사건임'이라고 태깅하는 컨텍스트 메뉴**가 없습니다.
* **개선안:**
1. 에디터에서 텍스트 선택 시 뜨는 플로팅 툴바(`editor-toolbar`)에 **[🏷️ 태그 추가]** 버튼 추가.
2. 클릭 시 '복선', '캐릭터 등장', '사망' 등 프리셋 태그 선택 팝업 노출.
3. 해당 정보를 백엔드의 `chunk_repo`나 별도 메타데이터 저장소에 `span` 정보와 함께 저장.




* **3. 정합성 오류의 '무시' vs '해결' UX 모호:**
* **현황:** `verdict-whitelist`("예외 처리")와 `verdict-ignore`("숨기기") 버튼이 있습니다. 사용자 입장에서 이 둘의 차이가 직관적이지 않을 수 있습니다.
* **개선안:**
* **오탐 신고(False Positive):** "이건 오류가 아님" -> 엔진 학습 또는 룰 수정 제안 (장기적).
* **의도된 설정 파괴:** "의도한 것임" -> 화이트리스트 + 이유 입력 팝업(UX 보강).
* **단순 무시:** "지금은 안 볼래" -> UI에서만 숨김.
* 버튼 명칭을 `[✅ 설정 수정]`, `[🛡️ 의도된 연출]`, `[👁️ 숨기기]` 등으로 구체화할 필요가 있습니다.




* **4. 에디터 가독성 설정 저장:**
* `localStorage`를 사용해 폰트/크기를 저장하는 것은 좋으나, 프로젝트별로 설정이 다를 수 있으므로(예: 동양풍 소설은 명조, SF는 고딕) **프로젝트 단위 설정 저장**도 고려해볼 만합니다.

### 1. 정합성 검사(nf_consistency) 분석 및 개선 제안

**분석 기준:** `user_request.md` (요구사항) vs. `modules/nf_consistency` (실제 구현)

#### ✅ 구현된 사항 (Positive)

1. **3단계 판정 파이프라인 (Layer 1~3):**
* 요구사항의 "명시 필드(나이/사망 등) 우선 검사"와 "모델(NLI) 위임" 구조가 `ConsistencyEngine`에 `_judge_with_fact_index`(Layer 1/2)와 `infer_nli_distribution`(Layer 3)으로 구현되어 있습니다.
* `RuleExtractor`를 통해 나이, 시간, 장소 등의 하드 제약(Hard Constraints)을 추출하는 로직이 정규식 기반으로 구현되어 있어 성능 최적화가 고려되었습니다.


2. **근거(Evidence) 기반 검증:**
* FTS(전문 검색)를 통해 근거 문장을 찾고(`EvidenceBuilder`), 이를 바탕으로 판정을 내리는 구조가 잡혀 있습니다.


3. **화이트리스트/예외처리:**
* `WhitelistService` 및 엔진 내 `excluded_self_evidence` 로직을 통해 사용자가 의도한 모순을 예외 처리할 수 있는 백엔드 메커니즘이 존재합니다.



#### ❌ 미비점 및 한계 (Gaps)

1. **서술 트릭/심리 묘사의 구분 부족 (요구사항: "암시/추정/심리/서술 트릭"은 Unknown 처리):**
* **현황:** 현재 구현은 주로 명시적 키워드(나이, 장소 등) 매칭과 NLI 모델 점수에 의존합니다. "거짓말"이나 "서술 트릭"을 문맥적으로 탐지하여 별도 분류하는 로직(예: 화자 신뢰도 분석)은 코드상 명확하지 않습니다.
* **위험:** 인물이 거짓말을 하는 상황에서 이를 '설정 오류'로 오판할 가능성이 높습니다.


2. **신뢰성 점수의 세분화 부족:**
* **요구사항:** 점수를 `FTS 매칭 강도 / 근거 / 확정근거 / 모델 추론 점수`로 분해하여 제공.
* **현황:** `VerdictLog`에 `confidence` 필드가 있으나, 사용자가 직관적으로 "왜 이 점수가 나왔는지" 알 수 있는 세부 구성 요소(Breakdown)가 UI나 응답 DTO에 충분히 노출되지 않고 있습니다.


3. **동적/잠정적 팩트(Draft Fact) 처리 미흡:**
* **현황:** 정합성 검사는 주로 '이미 DB에 저장된 스키마(Facts)'와 비교합니다. 사용자가 방금 작성했으나 아직 스키마로 승인되지 않은 내용과의 충돌을 실시간으로 잡아내는 데에는 한계가 있을 수 있습니다.



#### 💡 개선 제안

1. **화자/상황 기반 필터링 (Context Awareness):**
* 단순 문장 비교를 넘어, 해당 문장이 '대사(Dialogue)'인지 '지문(Narration)'인지 구분하는 플래그를 추가합니다. 대사일 경우 "등장인물의 주관적 주장(거짓말 가능성)"으로 간주하여 `UNKNOWN` 비중을 높이거나 `Potential Lie` 태그를 붙이는 로직을 추가합니다.


2. **신뢰도 점수 시각화 데이터 제공:**
* API 응답에 단순히 `OK/VIOLATE`만 주는 것이 아니라, `{"fts_score": 0.8, "nli_score": 0.2, "source_reliability": "high"}`와 같이 점수 구성요소를 리턴하여 UI에서 "근거는 확실하나 해석이 모호함" 등으로 툴팁을 띄울 수 있게 합니다.


3. **실시간 "Draft Context" 반영:**
* 편집 중인 문서 내에서 앞부분에 작성한 새로운 설정을 뒷부분 검사 시 임시 팩트로 메모리에 유지하는 `SessionContext`를 엔진에 주입하여, 저장 전이라도 문맥 내 모순을 잡도록 개선합니다.

### 2. UI/UX (HTML/Web Interface) 분석 및 개선 제안

**분석 기준:** `user_ui.html`, `debug_ui.html` (현 구현) vs. 제품 사용성

#### ✅ 구현된 사항 (Positive)

1. **역할 분리:** `debug_ui`는 개발자용(토큰 관리, 로우 레벨 호출), `user_ui`는 작가용(에디터 중심, 사이드바)으로 명확히 분리되었습니다.
2. **백그라운드 작업 가시화:** `SSE(Server-Sent Events)`를 통해 인덱싱, 정합성 검사 등의 진행 상황을 실시간으로 보여주는 구조(`jobs-widget`)가 잘 잡혀 있습니다.
3. **에디터 편의성:** 글자 크기, 자간/행간 설정 등 레이아웃 커스텀 기능이 포함되어 있습니다.

#### ❌ 미비점 및 한계 (Gaps)

1. **수동적인 정합성 피드백 루프:**
* **현황:** 정합성 검사 결과가 별도 패널이나 배지로만 뜹니다. 에디터 본문 내에서 오류가 있는 문장에 밑줄이 그어지고, **클릭 시 바로 "화이트리스트 추가"나 "수정 제안" 팝업이 뜨는 UX**가 미약합니다.
* **문제:** 사용자는 오류를 확인하기 위해 시선을 에디터에서 패널로 계속 옮겨야 합니다.


2. **인용(Reference) 연결성 부족:**
* **요구사항:** 위배 시 근거(어떤 문서/어떤 설정)를 경로 형태로 제시.
* **현황:** 결과 카드에 텍스트로 정보가 표시되지만, 클릭 시 해당 설정 문서의 해당 위치로 바로 점프하거나, 설정 문서를 팝업으로 띄워 비교하는 "Deep Linking" UX가 부족합니다.


3. **타임라인/관계도 시각화 부재:**
* **요구사항:** 시점(Timeline), 인물 관계 시각화.
* **현황:** `user_ui.html`의 사이드바는 단순 목록 형태입니다. 사건의 선후 관계나 인물 간의 관계를 직관적으로 보여주는 그래픽/타임라인 뷰가 없습니다.

## 2) 정합성 엔진(nf_consistency) “의도 반영도”와 치명 미비점

### 2-1. user_request 의도와 잘 맞는 부분(구조/정책)

* **Claim–Evidence–Verdict 강제**: 근거 없으면 UNKNOWN으로 떨어뜨리는 구조가 구현되어 있음.
* **UNKNOWN 사유 코드화**: `NO_EVIDENCE / AMBIGUOUS_ENTITY / SLOT_UNCOMPARABLE / CONFLICTING_EVIDENCE` 등을 저장.
* **Whitelist/Ignore 억제**: fingerprint 기반 반복 경고 억제 흐름 존재.
* **(옵션) 검증 루프**: UNKNOWN일 때 쿼리 변형으로 재검색 시도.

### 2-2. “신뢰도 점수”가 user_request 의도대로 동작하지 않을 가능성(정확성 문제)

현재 `ReliabilityBreakdown.fts_strength`에 **retrieval 결과의 raw score**가 그대로 들어가는데, 스코어 체계가 소스별로 다릅니다.

* FTS(BM25)는 **낮을수록 좋음**
* 벡터/기타는 **높을수록 좋음**

그런데 신뢰도 계산은 `1/(1+fts_strength)` 형태라서, **벡터 결과가 상위에 올라오면 점수가 오히려 떨어지는 역전 현상**이 발생할 수 있습니다.
→ user_request의 “FTS 매칭 강도/근거/확정근거/모델점수 분해” 중 **‘매칭 강도’ 축이 소스별로 일관되지 않음**.

**개선 방향(구체)**

* `fts_strength`를 raw가 아니라 **정규화된 [0,1] strength(높을수록 좋음)** 로 바꾸거나,
* `breakdown`에 `retrieval_source`, `retrieval_strength_norm`을 추가하고 UI에 그대로 노출.

### 2-3. Layer3(모델 기반) 승격/판정이 사실상 “작동 불능”일 가능성(confirmed 전파 부재)

엔진의 L3 승격 조건에 “confirmed evidence ≥ 2”가 들어가는데,

* `fts_search()`와 `vector_search()`가 반환하는 retrieval evidence는 **항상 `confirmed:false`**입니다.

즉, 실제 실행에서는 confirmed count가 0에 고정될 가능성이 높고, 그 결과:

* **L3 승격(UNKNOWN→OK)** 가 거의 발생하지 않음
* user_request에서 의도한 “근거 확정성 기반 신뢰도/승격” UX가 체감되지 않음

**개선 방향(구체)**

* `fts_search`에서 `tag_assignment.created_by==USER`가 겹치는 청크/스팬을 **confirmed:true**로 올리거나,
* “설정/인물 문서(SETTING/CHAR) + USER 태그 포함” 같은 규칙으로 confirmed를 부분 전파,
* 혹은 Evidence 테이블과 조인해 “이미 user-confirmed로 저장된 근거”를 재사용.

### 2-4. “의미 기반 정합성” 의도가 충분히 반영되지 않은 지점(범위/표현력 문제)

현재 구조는 강점이 “슬롯(나이/시간/장소/관계/…)" 기반 비교입니다. 이는 high precision에 유리하지만,

* **심리/서술 트릭/간접 떡밥 모순**처럼 user_request가 “unknown 처리 + 모델/휴리스틱 위임”을 말한 영역은,

  * (a) **클레임 자체를 충분히 생성/정형화하지 못하면** 검사 대상이 안 됨
  * (b) NLI도 현재는 “클레임 텍스트(종종 짧음) vs 상위 스니펫” 형태라서 의미 판독력이 제한됨

**개선 방향(구체)**

* NLI 입력을 `claim_text`가 아니라 **`segment_text(문장 전체)`**로 전환(또는 둘 다 평가)
* “슬롯 없는 문장”이라도 **retrieval + NLI로 ‘모순 가능성’만 UNKNOWN으로 표기**하는 별도 레인(낮은 우선순위) 추가
* 최소한 UI의 “딥 점검” 라벨은 현재 구현 범위(슬롯 위주)와 맞게 조정하지 않으면 UX 신뢰를 잃기 쉬움

## 3) GraphRAG(그래프 기반 검색/확장) 완성도와 병목

### 3-1. 구현은 존재하지만 “그래프가 비어” 실효가 약해질 위험

그래프 확장/리랭크는 `entity_mention_span`, `time_anchor`, `timeline_event`, `entity_alias`, `schema_facts(APPROVED)`를 기반으로 합니다.
문제는 기본 파이프라인(UI의 post-save 파이프라인)이:

* `INDEX_FTS`를 호출하긴 하지만, **entity/time grouping을 켜서 anchor/mention을 만들지 않음**
* 사용자 UI에서도 엔티티/타임라인 구축을 “명시적으로 트리거”하는 흐름이 없음

결과적으로 실사용에서는:

* 그래프 시드가 없어 `no_seeds`로 빠지며, graph_mode가 **사실상 no-op**이 될 가능성이 큼

**개선 방향(구체)**

* “그래프 인덱싱(엔티티 mention / time anchor / timeline)”을 별도 Job로 분리하고 UI에서 **명시적 실행 버튼** 제공
* 혹은 CONSISTENCY preflight에서 조건부로(예: graph_mode=auto이고 seeds가 부족할 때) grouping을 1회 수행

### 3-2. 그래프의 표현력 한계(고도화 타당성)

현재 그래프는 “같은 엔티티/시간 키를 공유하는 문서” 수준의 약한 연결입니다. 고도화를 한다면, 가장 위험한 실패 모드는:

* **엣지 생성 휴리스틱이 오염**되어 잘못된 문서를 계속 끌어오고,
* 정합성 엔진이 “근거 있음”으로 착각해 신뢰도/경고를 왜곡하는 것

따라서 고도화의 타당성은 “성능”보다 “오염 방지” 관점에서 봐야 합니다.

* 고도화가 필요하다면, **엣지에 ‘근거 타입/생성 경로/확신도’를 붙이고**, 확장 시에는 보수적으로 weight decay를 강하게 주는 쪽이 안전합니다.
* 클러스터링/임베딩 기반 엣지는 특히 오염 위험이 크므로, 최소한 **화이트리스트/승인 기반(confirmed)** 을 먼저 해결한 뒤 확장하는 순서가 안전합니다.

## 4) UI/UX 관점: 완성도/리스크(“동작 신뢰성” 중심)

### 4-1. “프로세스 분리로 UI 프리징 방지” 요구 미충족(구조 리스크)

오케스트레이터가 워커를 **스레드로 띄우는 구조**라서, heavy job이 메모리/CPU를 잠식하면:

* UI 서버(HTTP) 응답 지연/끊김
* SQLite contention으로 저장/조회 지연
* 최악에는 앱 전체 프리징

user_request가 말한 “인덱서/임베더 별도 프로세스, 죽여도 UI 유지”와 정면 충돌합니다.

### 4-2. API 계약 불일치로 UI 위젯이 깨질 가능성

* UI는 `GET /jobs?limit=5`를 호출하지만, 서버는 `/jobs`의 GET을 지원하지 않습니다(POST만).
  → Jobs 패널은 항상 실패할 가능성이 큼.
* Export: 워커는 파일을 만들지만, **Job 결과(result) 필드가 없고** 다운로드 엔드포인트도 없습니다.
  → “내보내기” 버튼이 사용자 입장에서 실패/무응답이 될 위험이 큼.

이런 “버튼은 있는데 실제로는 실패”는 UX 신뢰를 크게 떨어뜨립니다.

### 4-3. “변경 구간만 검사” 구현이 문서 규모가 커질수록 폭발할 수 있음

현재 변경 감지는 세그먼트를 `(start,end)`로 키잉하고 fingerprint를 비교합니다.
문서 앞부분에 글자를 삽입하면 뒤의 모든 세그먼트의 start/end가 밀려서:

* 실제로는 일부만 바꿨는데 **대량 세그먼트가 전부 changed로 인식**될 수 있음
* pending 큐가 커지고, 백그라운드 정합성 job이 계속 돌아 CPU/IO를 태울 수 있음

**개선 방향(구체)**

* “오프셋 기반” 대신 **내용 기반 LCS/디프**로 변경 구간만 추출
* 또는 세그먼트를 “순서+fingerprint”로 매칭해 이동/삽입에 강하게 만들기

### 4-4. “TIMELINE” 탭/문서 타입 설계가 혼재되어 런타임 오류 위험

UI 탭에 `TIMELINE`이 있으나 백엔드 `DocumentType`에는 TIMELINE이 없습니다.
탭이 “뷰 전환”이라면 문서 생성 버튼이 **비활성/가드**되어야 하는데, 현재 흐름에서는 잘못된 type을 POST할 가능성이 있습니다.

### 4-5. 기능 표기(라벨)와 실제 동작의 괴리

* “꼼꼼하게(심리/간접 떡밥 모순)”처럼 보이지만, 실제로는 extraction mode가 기본값이면 **슬롯 기반 검사에 머무를 가능성**이 큼.
* 사용자가 “딥 점검”을 눌렀는데 결과가 얕으면, 이후 전체 기능에 대한 신뢰가 급격히 무너집니다.

## 5) 개선 제안(우선순위: ‘동작 신뢰성’에 치명적인 것부터)

1. **confirmed 근거 전파(필수급)**

   * retrieval 결과의 evidence에 confirmed를 채우지 않으면, L3 승격/신뢰도 분해 설계가 무력화됩니다.

2. **신뢰도 점수의 소스-정합화(필수급)**

   * FTS/벡터 점수 체계를 섞어 쓰는 상태에서 raw를 넣으면 점수가 역전됩니다(특히 벡터 상위 시).

3. **UI-API 계약 불일치 제거(필수급)**

   * `/jobs` list, export 다운로드(또는 결과 반환) 없으면 UI 버튼/패널이 지속적으로 깨집니다.

4. **변경 구간 감지(오프셋 의존) 개선(대형 문서에서 필수급)**

   * “앞부분 편집 → 전체 재검사”는 실제 사용자 환경에서 가장 먼저 폭발합니다.

5. **GraphRAG가 실제로 쓰이게 하는 ‘그래프 인덱싱 트리거’ 제공(권장)**

   * 지금은 그래프가 비어 있을 가능성이 높아 고도화 이전에 “사용 가능 상태”부터 만드는 게 선행입니다.

6. **시간축/상태변화 표현(장기적으로 필수)**

   * 나이/관계/소속처럼 변하는 사실을 정적 fact로만 비교하면, 오탐을 whitelist로 땜질하게 되고 스키마가 오염됩니다.
