## 1) 현재 nf 구현에서 “의미 기반 정합성 + 위배구간 segmentation + heavy job 백그라운드” 의도 적용 여부(현상 기준)

### 1.1 UI에서 “변화가 거의 없는 텍스트 구간만” 자동 점검하도록 설계가 들어가 있음

* 툴바에 **consistency-status** 영역이 따로 있고(상태 노출 의도) 
* UI state에 **backgroundConsistencyTimer / backgroundConsistencyRunning / pendingConsistencySegments / lastSegmentFingerprints / lastVerdicts**가 존재해서 “백그라운드 점검 + 구간 단위 캐싱”을 전제로 함 
* 문장 단위로 쪼개고(문장부호/줄바꿈/소수점 예외 포함), 각 구간에 fingerprint를 붙인 뒤 
* **닫힌(closed) 구간만** + 이전 fingerprint와 달라진 구간만 모아 “변경된 문장만 점검”하려는 로직이 들어가 있음 

즉, user_request.md가 의도한 **“이미 확정된 구간에서 백그라운드로 정합성 검사 → UI에 위배 구간을 표시”** 하는 UX 방향 자체는 UI 코드에 반영되어 있습니다.

### 1.2 UI에 “위배/unknown 하이라이트”까지는 들어가 있음

* `::highlight(nf-violate)`, `::highlight(nf-unknown)` 및 fallback 클래스 스타일이 정의되어 있어(표시 레일 확보) 
* 다만 “정확히 어떤 span을 어떤 기준으로 highlight로 칠하는가”는 (파일 뒷부분의 JS에 있을 가능성이 크지만) 현재 보이는 부분만으로는 완결 여부를 단정하기 어렵습니다.

### 1.3 엔진 측 “의미 정보”는 **제한적으로** 쓰이고 있음 (완전한 임베딩 기반 의미검색은 아님)

* nf-consistency 엔진은 문장을 segment로 나눈 뒤 , extractor pipeline이 있으면 slot(나이/시간/장소/관계/소속 등)을 뽑아 claim을 구성합니다  
* 엔진 내부에 **triage / graph_mode / layer3 rerank / NLI verifier** 같은 “의미 기반 보강 옵션” 스위치가 존재합니다 
* UI도 graph_mode, layer3, verifier, triage, verification_loop 옵션을 노출하고 있습니다 
* 워커는 CONSISTENCY job의 결과 payload에 unknown reason counts, graph 적용 카운트, layer3 카운트 등을 포함하도록 이미 테스트가 잡혀 있습니다 

하지만 **“의미검색(Vector)” 자체는 아직 ‘토큰 오버랩 스텁’** 입니다. manifest에 “Vector search: token overlap for now.”가 박혀 있고 , 실제로는 토큰 교집합 기반 점수로 shard를 뒤져서 가져옵니다. 즉 “임베딩 기반 의미공간 검색”이 현재 구현의 기본 전제가 아닙니다.

### 1.4 heavy job을 UI 밖에서 돌리는 구조는 구현되어 있고, 성능 보호장치도 존재

* worker runner가 heavy job 타입을 `INDEX_VEC / CONSISTENCY / RETRIEVE_VEC`로 묶고, **동시 heavy job 개수를 설정(max_heavy_jobs)** 으로 제한합니다 
* 설정에는 `max_ram_mb / max_loaded_shards / max_heavy_jobs`가 있으며(평범한 PC 보호 전제) 
* 메모리 압력이 감지되면 worker가 job leasing 자체를 잠시 멈추는 형태로 “소프트 가드”가 동작합니다 
* 다만 TODO에 명시된 것처럼 CPU/PAUSE 등은 미구현이라(“멈췄다/기다리는 중”을 UI가 설명 못하면 UX가 무너짐)  이 부분은 UI/UX 관점에서 취약합니다.

## 2) “평범한 디바이스 + 800화 이상”을 전제로 한 최종 개선 전략 (UI/UX 중심, 성능 최우선)

아래는 **“글 쓰는 동안은 절대 느려지지 않게”** 만들기 위한 설계입니다. 핵심은 *foreground는 거의 공짜*, *background는 예산/범위/우선순위가 명확*해야 합니다.

---

## 2.1 UX 레이어를 3단으로 나눠 “항상 빠르게 보이는” 구조로 고정

### (A) Foreground(즉시 반응, 50~150ms 목표)

* 편집/스크롤/선택/저장은 **절대 heavy job과 결합하지 않음**
* 표시 가능한 것은:

  * “저장됨/저장 중” + “최근 점검 상태(최신/진행 중/대기)” 정도만 툴바에 표시 
* 여기서 중요한 UX 규칙:

  * **분석이 느려도 편집이 느려지면 실패** (대형 프로젝트에서 특히)

### (B) Background-light(자동, 변경된 닫힌 문장만)

UI는 이미 “닫힌 문장만, 바뀐 문장만” 모으는 로직이 있으므로  이를 **정식 정책**으로 격상:

* 트리거:

  * (1) 저장 직후 1~3초 debounce
  * (2) 타이핑 멈춤 2~5초 debounce
* 범위:

  * `pendingConsistencySegments`에 쌓인 닫힌 문장만
  * 여러 문장이 연속이면 **연속 구간으로 묶어** job payload의 `range`로 보냄(잡 수 폭증 방지)
* 옵션:

  * 기본은 “빠른 기본 점검” 프리셋(= layer3/verifier/triage/loop OFF, graph_mode는 auto 권장)
* 결과:

  * VIOLATE/UNKNOWN만 우측 패널과 하이라이트로 반영(OK는 숨김이 기본) — UI 토글도 이미 있음 

### (C) Background-deep(사용자 요청/스케줄, 비용 큰 의미추론)

* “지금 점검하기”는 deep를 허용하되, 디폴트는 light
* deep의 의미:

  * graph_mode(on/auto), layer3 rerank/NLI verifier, triage 등을 켜는 모드 
* 800화에서 deep를 자동으로 돌리면:

  * 배터리/발열/팬소음/작업지연/DB 팽창이 현실적으로 터집니다(최악 시 사용자 이탈)

---

## 2.2 잡(heavy job) UX: “돌고 있는지/왜 느린지/멈췄는지”를 UI가 설명해야 함

현재 NF는 `/jobs` + `/jobs/{id}/events` 스트리밍 계약을 갖고 , job event에는 progress/metrics/payload 확장이 가능합니다 .
여기서 UI/UX의 최종 형태는:

### (1) “작업 패널”을 우측 사이드바에 상시 제공

* TODO에도 “Job 상태/진행률/취소/재시도 UI”가 Must로 잡혀있는데 
* 현재 웹 UI에는 “작가 도우미(오류 점검)” 패널이 있으니  같은 공간에 “작업” 섹션을 추가하는 게 자연스럽습니다.

### (2) heavy job 동시 1개 정책을 **UI에 명시**

* worker는 heavy_types를 묶어 동시 실행을 제한합니다 
* 사용자에게는:

  * “현재 (벡터 인덱싱/점검) 작업이 진행 중이라 다음 점검은 대기”가 보여야 함
  * 그렇지 않으면 “버튼 눌렀는데 왜 아무 일도 안 생기지?”가 됩니다.

### (3) 메모리 압력/자원 보호로 “잠깐 멈춤”이 발생할 때 UI가 설명해야 함

* 지금은 메모리 압력이면 worker가 leasing을 멈출 뿐(상태/이벤트로 ‘멈춤’을 명확히 전달하지 않으면)  사용자 입장에서는 **무한 로딩/먹통**처럼 보일 수 있습니다.
* 해결 전략(UX 관점):

  * job 이벤트에 “PAUSED_DUE_TO_MEMORY_PRESSURE” 같은 WARN 이벤트를 1회라도 기록
  * 작업 패널에서 “자원 보호로 일시 중지(현재 RSS/제한)” 표시
* runner에는 rss_mb_peak 같은 표준 metric payload 구성 함수가 이미 있어 UI에 보여줄 재료가 있습니다 

---

## 2.3 “정합성 결과 표시” UX: 결과를 ‘이유’와 ‘근거’로 먼저 설명하고, 하이라이트는 그 다음

### (1) 결과 카드 구조를 고정

* 현재 시스템은 VerdictLog에 reliability breakdown(fts_strength/evidence_count/confirmed/model_score)을 저장하도록 설계되어 있고 
* 그리고 worker가 unknown reason counts까지 payload로 올릴 수 있게 되어 있습니다 
  → UI는 단순 “오류/정상”보다 아래를 기본으로 보여야 합니다.
* **왜 UNKNOWN인가** (NO_EVIDENCE / AMBIGUOUS_ENTITY / SLOT_UNCOMPARABLE / …) 
* **근거가 몇 개고(증거 수), 확정 근거가 몇 개인지**
* **근거가 약해서 모델이 켜져도 뒤집을 수 없는 정책인지**(evidence_required 원칙)

### (2) 하이라이트 정책(성능/정확성 동시 고려)

* contenteditable에서 DOM을 직접 감싸는 방식은 깨지기 쉬우므로, 지금처럼 `::highlight(...)`를 쓰는 방향이 상대적으로 안전합니다 
* 성능 정책:

  * 화면에 보이는 영역(뷰포트)만 highlight range를 적용(전체 문서 800화급에서는 필수)
  * VIOLATE > UNKNOWN 우선순위로 겹침 해결

### (3) “무시/화이트리스트”를 카드 단위로 바로 제공

* 반복 경고 억제는 ignore_item/whitelist_item + 엔진 연동이 이미 들어가 있는데, UI가 없으면 실사용성이 떨어집니다  
* 카드 버튼:

  * “이 경고 무시(이 문장)” → ignore 저장
  * “의도된 모순/거짓말(화이트리스트)” → whitelist 저장
* 이 2개가 없으면, 대형 소설에서 같은 경고가 계속 떠서 UX가 붕괴합니다.

## 2.4 800화 이상 스케일링: “범위 제한(Scoping)” 없이는 어떤 의미추론도 유지비가 감당 안 됨

### (1) ‘현재 문서 + 주변 문서’ 기본 스코프

* **전편(800화 전체)**을 기본 검색 공간으로 두면:

  * FTS는 그럭저럭 버티더라도,
  * graph/doc 후보 확장, vector shard 로딩, 모델 rerank/NLI는 비용이 폭발합니다.
* 따라서 기본 스코프를:

  * “현재 문서”
  * “설정/캐릭터/타임라인 문서(세계관 기준 문서)”
  * “인접 N편(예: ±10편)”
    로 고정하고, 전체 스캔은 ‘명시적’으로만 허용하는 게 안전합니다.

### (2) episode_id 매핑/전파를 **성능 기능으로 취급**

* TODO에도 “episode(n~m) 정의와 doc/스냅샷 매핑 규칙 확정”이 아직 TODO로 남아 있고 
* 그런데 retrieval 필터에는 episode가 이미 들어가 있습니다 
  → 800화에서 episode 스코프가 불안정하면 “의미 기반”이 아니라 그냥 “전역 검색”이 되어 성능이 망가집니다.
  **결론:** episode 매핑(번호/구간)이 확정되지 않으면, 대형 프로젝트 성능 최적화는 구조적으로 불가능합니다.

### (3) GraphRAG는 “품질”보다 “비용 절감용 prefilter”로 쓰는 게 타당

* 현재 graph는 entity/time/timeline 기반으로 doc 후보를 뽑고(최대 2-hop, doc_cap 제한)  
* graph를 materialize하면 doc_id 연결 테이블을 JSON으로 저장합니다 
* 또한 graph seed는 entity alias/terms로 query를 매칭하는데, 짧은 조사 처리까지 포함합니다 
  → 이 구조는 “대형 코퍼스에서 후보 doc_ids를 200개 이하로 줄여서” FTS/Vector를 때리도록 만드는 용도로는 합리적입니다.

**비관적 관점(중요):**

* entity/time anchor 자체가 오염되면(추출 오류/동명이인/별칭 누락) graph가 잘못된 doc을 우선시해서 **확신을 더하는 방향으로 편향**될 수 있습니다.
* 따라서 graph는:

  * `graph_mode=auto`가 기본(FTS가 약할 때만)
  * “근거 없는 결론”에는 절대 쓰지 않음(evidence_required 유지)
    이 안전합니다.

---

## 2.5 “의미 기반” 고도화(임베딩 포함): 지금 상태에서의 최우선은 **Vector를 ‘진짜 임베딩’으로 바꾸는 것**이지만, 자동 상시 실행은 금지해야 함

현재 vector는 토큰 오버랩 기반 스텁  이라서, “의미 기반”이라는 목표를 장기적으로 달성하려면 실제 임베딩 인덱스가 필요합니다.
다만 800화+온디바이스에서의 현실적 최적 형태는:

### (1) 임베딩은 “항상”이 아니라 “필요할 때만”

* FTS-first는 계약상도 고정되어 있고(동기 query는 fts만) 
* Vector 확장은 비동기 job(RETREIVE_VEC)로만 노출하는 게 원칙입니다 
  → 자동 백그라운드(light)에서는 **FTS-only**로 고정하고,
  “UNKNOWN이 많고 NO_EVIDENCE 비중이 높을 때” 같은 조건에서만 벡터/모델을 켜는 쪽이 안정적입니다.

### (2) BitNet 기반 임베딩: “타당성”은 조건부, 실무적으로는 리스크가 큼

BitNet류(초저비트/1-bit 계열)는 “모델 가중치/연산 효율” 목적에 강점이 있지만, **온디바이스 ‘문장 임베딩’ 품질/일관성**에서 바로 이득을 본다는 보장은 약합니다. 특히:

* **전용 임베딩 모델이 아닌 LLM을 bitnet으로 줄여 임베딩처럼 쓰면**

  * 임베딩 공간이 검색 친화적으로 정렬되어 있지 않아(contrastive로 훈련된 sentence embedding 대비) 검색 품질이 크게 흔들릴 수 있습니다.
* **커널/런타임 현실**: bit 연산이 이론상 싸도, CPU/일반 환경에서 최적화가 없으면 오히려 느릴 수 있습니다.
* **디버깅 비용**: 임베딩 검색 품질 저하는 UX에서 “근거 못 찾음(UNKNOWN)” 폭증으로 바로 체감됩니다.

따라서 “성능 최우선 + 평범한 디바이스” 조건에서는:

* bitnet 임베딩을 핵심 경로로 두기보다,
* **검증된 소형 sentence embedding + int8/4bit 양자화(ONNX 등)** 를 우선으로 두고,
* bitnet은 “향후 옵션/실험 경로”로 격리하는 편이 실패 확률이 낮습니다.

# 3) 현재 구현 기준 “미비점”과 “최악 시나리오” 중심 리스크 정리

## 3.1 UI/UX 미비점(성능 악화로 바로 이어짐)

* 프리셋 적용 시 `alert()` 기반 안내는 작업 흐름을 끊고, 대형 프로젝트에서 누적 스트레스를 만듭니다(토스트/상태바로 대체 권장). 
* UI 옵션 기본값(HTML checked)과 state 기본값이 불일치해 보입니다(예: layer3/loop)  

  * 이 경우 사용자는 “분명 껐는데 왜 무거운 점검이 돌지?” 같은 불신을 갖기 쉽습니다.
* heavy job 대기/자원압력으로 실행이 늦어질 때, UI가 이유를 설명하지 못하면 “먹통”으로 인식됩니다. (runner는 leasing만 멈출 수 있음) 

## 3.2 데이터/인덱스 측 최악 시나리오(800화에서 현실적으로 발생)

* **graph materialize의 크기 폭발**: entity_doc_ids/time_doc_ids/timeline_doc_ids가 커지면 JSON이 매우 커지고 로드가 느려집니다 

  * 해결책: doc_id 리스트를 압축/비트셋화하거나, SQLite 테이블/인덱스로 유지하고 필요한 부분만 조회하는 구조로 바꾸는 게 안전합니다.
* **vector shard 로딩 비용 폭발**: 현재는 샤드를 JSON으로 로드하고(캐시 제한은 max_loaded_shards 기반)  대형 프로젝트에서 I/O 병목이 큽니다.
* **episode 매핑이 불안정하면** 필터링이 무력화되어 전역 탐색이 기본이 되고, 결국 “자동 점검을 꺼야 하는 제품”이 됩니다. 

## 3.3 판정 품질 측 최악 시나리오(사용자 신뢰 붕괴)

* UNKNOWN이 많아지는 것은 괜찮지만(보수적 정책), 이유가 NO_EVIDENCE로 쌓이면 “시스템이 도움이 안 된다”로 읽힙니다. 
* graph/vector/model이 “근거 없는 확신”을 만들면 즉시 신뢰가 깨집니다. 이건 evidence_required 원칙으로 막고 있으나, UI가 그 원칙을 함께 보여주지 않으면 사용자 체감은 달라집니다.

# 결론(전략 요약)

* 현재 구현은 **“닫힌 문장만 자동 점검 + 결과 하이라이트”** 라는 UX 골격이 이미 들어가 있고 ,
* 워커는 **heavy job 동시성/메모리 보호**까지 갖추고 있어 , “평범한 PC에서의 안정성” 방향은 맞습니다.
* 다만 “의미 기반”을 실제로 끌어올리려면, 현재의 token-overlap vector  를 그대로 두고는 한계가 명확하며, 800화급에서는 **스코프(episode/주변/설정문서) 제한 + auto(light)와 deep의 분리 + job 상태 설명 UX**가 먼저 고정돼야 성능과 신뢰를 동시에 지킬 수 있습니다.



## 1) 임베딩을 BitNet 기반으로 수행하는 것의 타당성(비판적 검토)

### 1.1 “BitNet 임베딩”이 의미하는 바부터 정리

BitNet은 본질적으로 **Transformer의 선형층(Linear) 가중치를 1-bit/저비트로 근사**해 추론·메모리 비용을 줄이려는 계열입니다. “임베딩을 BitNet으로 수행”은 보통 아래 2가지 중 하나입니다.

1. **임베딩 전용 모델(문장/문서 임베딩 모델)을 BitNet/저비트로 양자화/대체**해서 온디바이스에서 굴린다
2. **현재 LLM(온디바이스) 내부의 hidden representation을 임베딩처럼 재활용**한다(예: last hidden pooling)

둘은 타당성/리스크가 다릅니다.

---

### 1.2 타당한 경우(제한적)

#### A. “임베딩 전용 모델”을 **저비트**로 가져가고 싶을 때

* 온디바이스에서 임베딩을 돌려야 하고, 배터리/지연이 빡빡하면 **저비트 임베딩 모델**은 타당합니다.
* 단, BitNet이 “임베딩 품질을 유지한 채” 최선인지가 문제입니다. 실무적으로는 BitNet보다 **검증된 8-bit/4-bit 양자화(INT8/INT4, GPTQ/AWQ류)**가 도구/안정성이 높고, 품질 손실도 예측 가능성이 큽니다.

#### B. 임베딩 품질 요구가 낮고, “후보군 좁히기” 수준이면

* nf의 정합성은 “판정”이 아니라 “근거 후보 검색/리랭크”에 임베딩을 쓰는 쪽이 UX 안전합니다. 이 경우 약간의 임베딩 열화가 치명적이지 않을 수 있습니다.

---

### 1.3 위험/비타당 케이스(현재 nf 목표에 더 중요)

#### 1) **정합성 판정(위배/비위배) 자체를 임베딩 유사도로 때우는 순간** UX가 망가질 확률이 큼

* 임베딩은 “유사함”을 주지만 “모순/불일치”를 안정적으로 주지 않습니다.
* 특히 소설/작문에서는 **유사해도 모순인 경우(시간/관계/속성 값 충돌)**가 많아서, 임베딩 기반 판정은 오탐/미탐이 동시에 커집니다.
* nf의 설계 방향이 `Claim–Evidence–Verdict`로 고정된 것도 이 리스크를 피하려는 쪽에 가깝습니다. 

#### 2) BitNet/극저비트는 “품질 저하의 형태”가 더 난감할 수 있음

* 유사도 검색에서 가장 위험한 건 “조용히 틀리는 것”입니다.
* 극저비트는 도메인/문체/고유명사에 민감해져서 **근거를 엉뚱한 곳에서 가져오는** 케이스가 늘면, 사용자는 “근거가 있는데도 틀린 판단”을 경험합니다(신뢰 붕괴).

#### 3) nf 현재 구현은 “의미 임베딩” 이전에 해결해야 할 UX/정확도 병목이 존재

* 현재 Vector 검색이 실제 임베딩이 아니라 **token overlap 스텁**이고 , 문서에도 스텁임이 명시되어 있습니다 .
* 이 상태에서 BitNet 임베딩을 넣는 건 “좋은 후보군을 만들기 위한 파이프라인” 자체가 아직 정리되지 않은 상태에서 최적화를 하는 셈이라, 투자 대비 효과가 낮을 가능성이 큽니다.

**결론:**

* “온디바이스 임베딩” 자체는 타당할 수 있으나, **BitNet을 최우선 선택지로 두는 건 리스크가 큽니다.**
* nf 관점에서 임베딩은 (1) 후보 검색 보강용, (2) 비결정적 신호는 판정이 아니라 breakdown/추천에만 반영(과신 방지)하는 것이 UX적으로 안전합니다.

## 2) UI/UX 관점에서 nf 고도화 방안 + 미비점 개선안(구체)

아래는 “현재 구현의 구조(FTS + optional vector + optional GraphRAG)”를 유지하면서, 사용자 체감(신뢰·설명가능·지연)을 개선하는 방향입니다.

---

### 2.1 가장 큰 미비점 1: “스키마 경로(tag_path) 기반 설명”이 UI에 제대로 안 올라옴

* TODO에 **위배 시 스키마 경로 직접 표기 미구현**이 명시되어 있습니다. 
* `user_request.md`는 태그 경로(설정/인물/속성 등)를 핵심 UX로 원합니다. 

#### 개선안(UX 스펙)

1. 결과 패널을 “위배 카드” 단위로 고정:

   * **문장 하이라이트(Span)**
   * **위배 타입**: (속성 값 충돌 / 시간 역행 / 관계 모순 / 금지 설정 위반 등)
   * **기준 경로**: `tag_path`(예: `characters/protagonist/age`)
   * **근거 인용**: `doc_id + section_path + span + snippet`
2. “왜 위배인지”는 Graph가 아니라 **경로 기반 비교**로 1줄 요약:

   * 예: `age: 17(승인 설정) vs 19(본문 언급)`

---

### 2.2 가장 큰 미비점 2: segmentation 품질이 작문 UX에 부족

* 문서에 “문장/절 segmentation 개선 필요”가 반복적으로 나옵니다.  
* 지금은 구두점/개행 기반이라  대사/따옴표/괄호/한국어 종결 표현에서 UX가 깨질 가능성이 큽니다.

#### 개선안(UX 스펙)

1. 세그먼트 단위를 “문장”만이 아니라 “문단/대사 턴”도 지원:

   * 대사(“ ”)는 한 턴을 하나의 세그먼트로 묶기
   * 문단은 기본 접힘 단위로 사용
2. 하이라이트가 너무 잘게 쪼개지면 “피드백 폭격”이 됩니다.

   * 연속 위배 span은 자동 병합(merge spans)해서 1카드로 보여주기
3. 검사 트리거:

   * “작성 중 실시간”이 아니라, 사용자 의도대로 **변화가 거의 없는 구간에서 백그라운드 검사**가 기본.
   * UI는 “검사 중/완료/오류”만 가볍게 보여주고, 결과는 사용자가 열람할 때만 펼치기.

---

### 2.3 가장 큰 미비점 3: Vector가 “의미”가 아니라 토큰 겹침 스텁

* `vector_search`가 token overlap 기반입니다. 
* 즉 “의미 기반 정합성”이 아니라 “단어 비슷함” 수준입니다.

#### 개선안(아키텍처는 최소 변경)

1. `vector_search()`를 “백엔드 플러그인”으로 분리:

   * `TokenOverlapBackend`(현재)
   * `EmbeddingBackend`(추가)
2. 임베딩을 도입하더라도, 역할은 **후보 검색 보강**으로 제한:

   * 판정은 `FTS 근거 + 스키마 비교`가 주도
   * 임베딩은 “근거 후보를 더 잘 찾는” 용도(UX 안전)
3. 임베딩 기반 결과에는 항상 **근거 출처(인용) 강제**:

   * `user_request.md`의 “근거 없으면 unknown” 원칙을 유지 

> 여기서 BitNet은 “EmbeddingBackend의 구현 옵션 중 하나”로 두되, 기본은 더 안정적인 저비트(예: INT8/4)로 시작하는 편이 실패 비용이 낮습니다.

---

### 2.4 GraphRAG 고도화: “판정”이 아니라 “설명/탐색 UX”에 투자해야 타당

현재 GraphRAG는 기본 off, rerank 옵션형입니다. 

#### 개선안(UX에 직접 이득이 있는 방향)

1. GraphRAG 결과를 사용자에게 “그래프”로 보여주지 말고, **근거 연결(why these docs)**로 보여주기:

   * seed_docs → expanded_docs를 “연결 이유(공유 엔티티/시간/장소)”로 텍스트로 요약
2. GraphRAG 실행은 백그라운드:

   * “상세 분석(느리지만 풍부)” 버튼에서만 켜기
3. Graph 품질이 낮을 때 UX 방어:

   * `expanded_docs`가 근거 인용을 못 만들면 “확장 실패/근거 부족”으로 표시하고 자동으로 영향도를 낮춤

---

### 2.5 화이트리스트/ignore UI가 없으면 오탐이 누적되어 제품이 망가짐

* TODO에 “화이트리스트/ignore는 엔진에 있으나 제품 UI는 미구현”이 명시됩니다. 

#### 개선안(UX 스펙)

* 위배 카드마다:

  * “이 문장 무시(이번만)”
  * “이 규칙/경로 무시(프로젝트 설정)”
  * “이 엔티티 표기 변형 허용(별칭 등록)”
* 작문 보조에서는 “오탐 관리 UI”가 없으면 결국 사용자가 기능을 꺼버립니다.

## 3) 최종 제안(요청사항에 대한 결론)

1. **BitNet 임베딩 채택 타당성**

* 온디바이스 임베딩 자체는 타당하지만, BitNet을 최우선으로 두면
  **조용한 품질 저하(근거 오염) → 신뢰 붕괴** 위험이 큽니다.
* nf에서는 임베딩을 **판정 신호가 아니라 후보 검색 보강 신호**로 제한하는 게 UX적으로 더 안전합니다. 
* 따라서 “BitNet 기반 임베딩”은 **옵션(성능/전력 최적화 단계)**로 두고, 먼저 “임베딩 백엔드 플러그인화 + 근거 강제”를 완성하는 편이 합리적입니다.

2. **UI/UX 관점 고도화 우선순위(ROI 순)**

* (1) **스키마 경로(tag_path) 기반 설명/표시**(현재 미구현) 
* (2) **segmentation 품질 개선**(문서에 개선 필요 반복) 
* (3) Vector를 “진짜 임베딩”으로 교체하되 역할은 후보 보강으로 제한 
* (4) GraphRAG는 “설명/탐색 UX”를 강화하는 방향으로만 고도화(기본 off 유지) 
* (5) **화이트리스트/ignore UI**로 오탐 누적을 방지 






## 1) 현재 nf 구현에서 “의미 기반 정합성 + 위배구간 segmentation + heavy job 백그라운드” 의도 적용 여부(현상 기준)

### 1.1 UI에서 “변화가 거의 없는 텍스트 구간만” 자동 점검하도록 설계가 들어가 있음

* 툴바에 **consistency-status** 영역이 따로 있고(상태 노출 의도) 
* UI state에 **backgroundConsistencyTimer / backgroundConsistencyRunning / pendingConsistencySegments / lastSegmentFingerprints / lastVerdicts**가 존재해서 “백그라운드 점검 + 구간 단위 캐싱”을 전제로 함 
* 문장 단위로 쪼개고(문장부호/줄바꿈/소수점 예외 포함), 각 구간에 fingerprint를 붙인 뒤 
* **닫힌(closed) 구간만** + 이전 fingerprint와 달라진 구간만 모아 “변경된 문장만 점검”하려는 로직이 들어가 있음 

즉, user_request.md가 의도한 **“이미 확정된 구간에서 백그라운드로 정합성 검사 → UI에 위배 구간을 표시”** 하는 UX 방향 자체는 UI 코드에 반영되어 있습니다.

### 1.2 UI에 “위배/unknown 하이라이트”까지는 들어가 있음

* `::highlight(nf-violate)`, `::highlight(nf-unknown)` 및 fallback 클래스 스타일이 정의되어 있어(표시 레일 확보) 
* 다만 “정확히 어떤 span을 어떤 기준으로 highlight로 칠하는가”는 (파일 뒷부분의 JS에 있을 가능성이 크지만) 현재 보이는 부분만으로는 완결 여부를 단정하기 어렵습니다.

### 1.3 엔진 측 “의미 정보”는 **제한적으로** 쓰이고 있음 (완전한 임베딩 기반 의미검색은 아님)

* nf-consistency 엔진은 문장을 segment로 나눈 뒤 , extractor pipeline이 있으면 slot(나이/시간/장소/관계/소속 등)을 뽑아 claim을 구성합니다  
* 엔진 내부에 **triage / graph_mode / layer3 rerank / NLI verifier** 같은 “의미 기반 보강 옵션” 스위치가 존재합니다 
* UI도 graph_mode, layer3, verifier, triage, verification_loop 옵션을 노출하고 있습니다 
* 워커는 CONSISTENCY job의 결과 payload에 unknown reason counts, graph 적용 카운트, layer3 카운트 등을 포함하도록 이미 테스트가 잡혀 있습니다 

하지만 **“의미검색(Vector)” 자체는 아직 ‘토큰 오버랩 스텁’** 입니다. manifest에 “Vector search: token overlap for now.”가 박혀 있고 , 실제로는 토큰 교집합 기반 점수로 shard를 뒤져서 가져옵니다. 즉 “임베딩 기반 의미공간 검색”이 현재 구현의 기본 전제가 아닙니다.

### 1.4 heavy job을 UI 밖에서 돌리는 구조는 구현되어 있고, 성능 보호장치도 존재

* worker runner가 heavy job 타입을 `INDEX_VEC / CONSISTENCY / RETRIEVE_VEC`로 묶고, **동시 heavy job 개수를 설정(max_heavy_jobs)** 으로 제한합니다 
* 설정에는 `max_ram_mb / max_loaded_shards / max_heavy_jobs`가 있으며(평범한 PC 보호 전제) 
* 메모리 압력이 감지되면 worker가 job leasing 자체를 잠시 멈추는 형태로 “소프트 가드”가 동작합니다 
* 다만 TODO에 명시된 것처럼 CPU/PAUSE 등은 미구현이라(“멈췄다/기다리는 중”을 UI가 설명 못하면 UX가 무너짐)  이 부분은 UI/UX 관점에서 취약합니다.

---

## 2) “평범한 디바이스 + 800화 이상”을 전제로 한 최종 개선 전략 (UI/UX 중심, 성능 최우선)

아래는 **“글 쓰는 동안은 절대 느려지지 않게”** 만들기 위한 설계입니다. 핵심은 *foreground는 거의 공짜*, *background는 예산/범위/우선순위가 명확*해야 합니다.

---

## 2.1 UX 레이어를 3단으로 나눠 “항상 빠르게 보이는” 구조로 고정

### (A) Foreground(즉시 반응, 50~150ms 목표)

* 편집/스크롤/선택/저장은 **절대 heavy job과 결합하지 않음**
* 표시 가능한 것은:

  * “저장됨/저장 중” + “최근 점검 상태(최신/진행 중/대기)” 정도만 툴바에 표시 
* 여기서 중요한 UX 규칙:

  * **분석이 느려도 편집이 느려지면 실패** (대형 프로젝트에서 특히)

### (B) Background-light(자동, 변경된 닫힌 문장만)

UI는 이미 “닫힌 문장만, 바뀐 문장만” 모으는 로직이 있으므로  이를 **정식 정책**으로 격상:

* 트리거:

  * (1) 저장 직후 1~3초 debounce
  * (2) 타이핑 멈춤 2~5초 debounce
* 범위:

  * `pendingConsistencySegments`에 쌓인 닫힌 문장만
  * 여러 문장이 연속이면 **연속 구간으로 묶어** job payload의 `range`로 보냄(잡 수 폭증 방지)
* 옵션:

  * 기본은 “빠른 기본 점검” 프리셋(= layer3/verifier/triage/loop OFF, graph_mode는 auto 권장)
* 결과:

  * VIOLATE/UNKNOWN만 우측 패널과 하이라이트로 반영(OK는 숨김이 기본) — UI 토글도 이미 있음 

### (C) Background-deep(사용자 요청/스케줄, 비용 큰 의미추론)

* “지금 점검하기”는 deep를 허용하되, 디폴트는 light
* deep의 의미:

  * graph_mode(on/auto), layer3 rerank/NLI verifier, triage 등을 켜는 모드 
* 800화에서 deep를 자동으로 돌리면:

  * 배터리/발열/팬소음/작업지연/DB 팽창이 현실적으로 터집니다(최악 시 사용자 이탈)

---

## 2.2 잡(heavy job) UX: “돌고 있는지/왜 느린지/멈췄는지”를 UI가 설명해야 함

현재 NF는 `/jobs` + `/jobs/{id}/events` 스트리밍 계약을 갖고 , job event에는 progress/metrics/payload 확장이 가능합니다 .
여기서 UI/UX의 최종 형태는:

### (1) “작업 패널”을 우측 사이드바에 상시 제공

* TODO에도 “Job 상태/진행률/취소/재시도 UI”가 Must로 잡혀있는데 
* 현재 웹 UI에는 “작가 도우미(오류 점검)” 패널이 있으니  같은 공간에 “작업” 섹션을 추가하는 게 자연스럽습니다.

### (2) heavy job 동시 1개 정책을 **UI에 명시**

* worker는 heavy_types를 묶어 동시 실행을 제한합니다 
* 사용자에게는:

  * “현재 (벡터 인덱싱/점검) 작업이 진행 중이라 다음 점검은 대기”가 보여야 함
  * 그렇지 않으면 “버튼 눌렀는데 왜 아무 일도 안 생기지?”가 됩니다.

### (3) 메모리 압력/자원 보호로 “잠깐 멈춤”이 발생할 때 UI가 설명해야 함

* 지금은 메모리 압력이면 worker가 leasing을 멈출 뿐(상태/이벤트로 ‘멈춤’을 명확히 전달하지 않으면)  사용자 입장에서는 **무한 로딩/먹통**처럼 보일 수 있습니다.
* 해결 전략(UX 관점):

  * job 이벤트에 “PAUSED_DUE_TO_MEMORY_PRESSURE” 같은 WARN 이벤트를 1회라도 기록
  * 작업 패널에서 “자원 보호로 일시 중지(현재 RSS/제한)” 표시
* runner에는 rss_mb_peak 같은 표준 metric payload 구성 함수가 이미 있어 UI에 보여줄 재료가 있습니다 

---

## 2.3 “정합성 결과 표시” UX: 결과를 ‘이유’와 ‘근거’로 먼저 설명하고, 하이라이트는 그 다음

### (1) 결과 카드 구조를 고정

* 현재 시스템은 VerdictLog에 reliability breakdown(fts_strength/evidence_count/confirmed/model_score)을 저장하도록 설계되어 있고 
* 그리고 worker가 unknown reason counts까지 payload로 올릴 수 있게 되어 있습니다 
  → UI는 단순 “오류/정상”보다 아래를 기본으로 보여야 합니다.
* **왜 UNKNOWN인가** (NO_EVIDENCE / AMBIGUOUS_ENTITY / SLOT_UNCOMPARABLE / …) 
* **근거가 몇 개고(증거 수), 확정 근거가 몇 개인지**
* **근거가 약해서 모델이 켜져도 뒤집을 수 없는 정책인지**(evidence_required 원칙)

### (2) 하이라이트 정책(성능/정확성 동시 고려)

* contenteditable에서 DOM을 직접 감싸는 방식은 깨지기 쉬우므로, 지금처럼 `::highlight(...)`를 쓰는 방향이 상대적으로 안전합니다 
* 성능 정책:

  * 화면에 보이는 영역(뷰포트)만 highlight range를 적용(전체 문서 800화급에서는 필수)
  * VIOLATE > UNKNOWN 우선순위로 겹침 해결

### (3) “무시/화이트리스트”를 카드 단위로 바로 제공

* 반복 경고 억제는 ignore_item/whitelist_item + 엔진 연동이 이미 들어가 있는데, UI가 없으면 실사용성이 떨어집니다  
* 카드 버튼:

  * “이 경고 무시(이 문장)” → ignore 저장
  * “의도된 모순/거짓말(화이트리스트)” → whitelist 저장
* 이 2개가 없으면, 대형 소설에서 같은 경고가 계속 떠서 UX가 붕괴합니다.

---

## 2.4 800화 이상 스케일링: “범위 제한(Scoping)” 없이는 어떤 의미추론도 유지비가 감당 안 됨

### (1) ‘현재 문서 + 주변 문서’ 기본 스코프

* **전편(800화 전체)**을 기본 검색 공간으로 두면:

  * FTS는 그럭저럭 버티더라도,
  * graph/doc 후보 확장, vector shard 로딩, 모델 rerank/NLI는 비용이 폭발합니다.
* 따라서 기본 스코프를:

  * “현재 문서”
  * “설정/캐릭터/타임라인 문서(세계관 기준 문서)”
  * “인접 N편(예: ±10편)”
    로 고정하고, 전체 스캔은 ‘명시적’으로만 허용하는 게 안전합니다.

### (2) episode_id 매핑/전파를 **성능 기능으로 취급**

* TODO에도 “episode(n~m) 정의와 doc/스냅샷 매핑 규칙 확정”이 아직 TODO로 남아 있고 
* 그런데 retrieval 필터에는 episode가 이미 들어가 있습니다 
  → 800화에서 episode 스코프가 불안정하면 “의미 기반”이 아니라 그냥 “전역 검색”이 되어 성능이 망가집니다.
  **결론:** episode 매핑(번호/구간)이 확정되지 않으면, 대형 프로젝트 성능 최적화는 구조적으로 불가능합니다.

### (3) GraphRAG는 “품질”보다 “비용 절감용 prefilter”로 쓰는 게 타당

* 현재 graph는 entity/time/timeline 기반으로 doc 후보를 뽑고(최대 2-hop, doc_cap 제한)  
* graph를 materialize하면 doc_id 연결 테이블을 JSON으로 저장합니다 
* 또한 graph seed는 entity alias/terms로 query를 매칭하는데, 짧은 조사 처리까지 포함합니다 
  → 이 구조는 “대형 코퍼스에서 후보 doc_ids를 200개 이하로 줄여서” FTS/Vector를 때리도록 만드는 용도로는 합리적입니다.

**비관적 관점(중요):**

* entity/time anchor 자체가 오염되면(추출 오류/동명이인/별칭 누락) graph가 잘못된 doc을 우선시해서 **확신을 더하는 방향으로 편향**될 수 있습니다.
* 따라서 graph는:

  * `graph_mode=auto`가 기본(FTS가 약할 때만)
  * “근거 없는 결론”에는 절대 쓰지 않음(evidence_required 유지)
    이 안전합니다.

---

## 2.5 “의미 기반” 고도화(임베딩 포함): 지금 상태에서의 최우선은 **Vector를 ‘진짜 임베딩’으로 바꾸는 것**이지만, 자동 상시 실행은 금지해야 함

현재 vector는 토큰 오버랩 기반 스텁  이라서, “의미 기반”이라는 목표를 장기적으로 달성하려면 실제 임베딩 인덱스가 필요합니다.
다만 800화+온디바이스에서의 현실적 최적 형태는:

### (1) 임베딩은 “항상”이 아니라 “필요할 때만”

* FTS-first는 계약상도 고정되어 있고(동기 query는 fts만) 
* Vector 확장은 비동기 job(RETREIVE_VEC)로만 노출하는 게 원칙입니다 
  → 자동 백그라운드(light)에서는 **FTS-only**로 고정하고,
  “UNKNOWN이 많고 NO_EVIDENCE 비중이 높을 때” 같은 조건에서만 벡터/모델을 켜는 쪽이 안정적입니다.

### (2) BitNet 기반 임베딩: “타당성”은 조건부, 실무적으로는 리스크가 큼

BitNet류(초저비트/1-bit 계열)는 “모델 가중치/연산 효율” 목적에 강점이 있지만, **온디바이스 ‘문장 임베딩’ 품질/일관성**에서 바로 이득을 본다는 보장은 약합니다. 특히:

* **전용 임베딩 모델이 아닌 LLM을 bitnet으로 줄여 임베딩처럼 쓰면**

  * 임베딩 공간이 검색 친화적으로 정렬되어 있지 않아(contrastive로 훈련된 sentence embedding 대비) 검색 품질이 크게 흔들릴 수 있습니다.
* **커널/런타임 현실**: bit 연산이 이론상 싸도, CPU/일반 환경에서 최적화가 없으면 오히려 느릴 수 있습니다.
* **디버깅 비용**: 임베딩 검색 품질 저하는 UX에서 “근거 못 찾음(UNKNOWN)” 폭증으로 바로 체감됩니다.

따라서 “성능 최우선 + 평범한 디바이스” 조건에서는:

* bitnet 임베딩을 핵심 경로로 두기보다,
* **검증된 소형 sentence embedding + int8/4bit 양자화(ONNX 등)** 를 우선으로 두고,
* bitnet은 “향후 옵션/실험 경로”로 격리하는 편이 실패 확률이 낮습니다.

# 3) 현재 구현 기준 “미비점”과 “최악 시나리오” 중심 리스크 정리

## 3.1 UI/UX 미비점(성능 악화로 바로 이어짐)

* 프리셋 적용 시 `alert()` 기반 안내는 작업 흐름을 끊고, 대형 프로젝트에서 누적 스트레스를 만듭니다(토스트/상태바로 대체 권장). 
* UI 옵션 기본값(HTML checked)과 state 기본값이 불일치해 보입니다(예: layer3/loop)  

  * 이 경우 사용자는 “분명 껐는데 왜 무거운 점검이 돌지?” 같은 불신을 갖기 쉽습니다.
* heavy job 대기/자원압력으로 실행이 늦어질 때, UI가 이유를 설명하지 못하면 “먹통”으로 인식됩니다. (runner는 leasing만 멈출 수 있음) 

## 3.2 데이터/인덱스 측 최악 시나리오(800화에서 현실적으로 발생)

* **graph materialize의 크기 폭발**: entity_doc_ids/time_doc_ids/timeline_doc_ids가 커지면 JSON이 매우 커지고 로드가 느려집니다 

  * 해결책: doc_id 리스트를 압축/비트셋화하거나, SQLite 테이블/인덱스로 유지하고 필요한 부분만 조회하는 구조로 바꾸는 게 안전합니다.
* **vector shard 로딩 비용 폭발**: 현재는 샤드를 JSON으로 로드하고(캐시 제한은 max_loaded_shards 기반)  대형 프로젝트에서 I/O 병목이 큽니다.
* **episode 매핑이 불안정하면** 필터링이 무력화되어 전역 탐색이 기본이 되고, 결국 “자동 점검을 꺼야 하는 제품”이 됩니다. 

## 3.3 판정 품질 측 최악 시나리오(사용자 신뢰 붕괴)

* UNKNOWN이 많아지는 것은 괜찮지만(보수적 정책), 이유가 NO_EVIDENCE로 쌓이면 “시스템이 도움이 안 된다”로 읽힙니다. 
* graph/vector/model이 “근거 없는 확신”을 만들면 즉시 신뢰가 깨집니다. 이건 evidence_required 원칙으로 막고 있으나, UI가 그 원칙을 함께 보여주지 않으면 사용자 체감은 달라집니다.

# 결론(전략 요약)

* 현재 구현은 **“닫힌 문장만 자동 점검 + 결과 하이라이트”** 라는 UX 골격이 이미 들어가 있고 ,
* 워커는 **heavy job 동시성/메모리 보호**까지 갖추고 있어 , “평범한 PC에서의 안정성” 방향은 맞습니다.
* 다만 “의미 기반”을 실제로 끌어올리려면, 현재의 token-overlap vector  를 그대로 두고는 한계가 명확하며, 800화급에서는 **스코프(episode/주변/설정문서) 제한 + auto(light)와 deep의 분리 + job 상태 설명 UX**가 먼저 고정돼야 성능과 신뢰를 동시에 지킬 수 있습니다.
