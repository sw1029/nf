제공된 요구사항 정의서와 UI/UX 구현 코드(HTML, CSS, JS)를 바탕으로, 요구사항 반영 현황 및 UI/UX 관점에서의 미비점과 비관적 발생 가능 시나리오를 검토한 결과입니다.

### 1. 요구사항 반영 현황

* **문서 분류 및 레이아웃 설정**: 본문, 설정, 구상, 타임라인 탭이 분리되어 있으며, 글자 크기, 줄 간격, 자간 등을 조정하는 `config-panel`이 구현되어 있습니다.
* **정합성 위배 처리 및 예외 관리**: `action-popover`를 통해 정합성 모순 감지 시 '의도됨(whitelist)' 및 '무시(ignore)' 액션을 수행할 수 있는 인터페이스가 적용되었습니다.
* **문법 교정 및 개선 제안**: 도우미 패널 내 'PROPOSE' 탭에서 오탈자, 자연스러운 윤문, 극적인 묘사 등 교정 강도를 설정하여 API를 호출할 수 있습니다.
* **룰베이스 사용자 태깅**: 텍스트 드래그 시 표시되는 `inline-tag-widget`을 통해 인물, 사건, 복선 등 룰베이스 태깅 및 메모 부여 기능이 반영되었습니다.
* **데이터 추출 및 API 설정**: txt, docx 형식의 내보내기가 모달을 통해 제공되며, 전역 설정 팝업에서 사용자가 API 키 및 사용할 AI 모델을 직접 기재할 수 있도록 구성되었습니다.

### 2. UI/UX 미비점 및 개선 필요 영역

* **점수 분해 정보 노출 부재**: 요구사항에는 신뢰성 점수의 근거를 FTS 매칭 강도, 확정 근거, 모델 추론 점수 등으로 분해하여 제공하도록 명시되어 있으나, 현재 UI는 `reliability_overall`이라는 단일 신뢰도 수치만을 노출하고 있어 사용자가 판정의 세부 근거를 직관적으로 납득하기 어렵습니다.
* **타임라인 및 인물 매핑의 시각화 부족**: 특정 시점이나 인물을 기준으로 그룹을 구성하고 필터링하는 입력창(`consistency-filter-entity` 등)은 존재하나, 텍스트의 Span 범위와 타임라인 인덱스(`timeline_idx`)를 사용자가 시각적으로 직접 매핑하고 확인하는 상호작용 UI가 구현되어 있지 않습니다.
* **예외 처리(화이트리스트) 통합 관리 UI 누락**: 모순으로 지적된 부분을 예외 처리할 수는 있으나, 사용자가 승인한 화이트리스트 목록을 사후에 일괄적으로 조회, 수정, 취소할 수 있는 대시보드나 패널이 존재하지 않습니다.

### 3. 비관적 검토 (최악의 상황 가정)

* **DOM 노드 과다로 인한 치명적 UI 프리징**: 텍스트 입력 및 붙여넣기 시 `paginateText` 함수가 메인 스레드에서 전체 텍스트의 높이를 동기적으로 측정하여 페이지 분할을 수행합니다. 1000화 이상의 고용량 텍스트나 무수히 많은 인라인 태그(`inline-tag-marked`)가 존재할 경우, 레이아웃 재계산(Reflow) 비용이 기하급수적으로 증가하여 브라우저 탭 크래시나 심각한 입력 지연이 발생할 수 있습니다.
* **비동기 작업 상태 동기화 실패 및 영구 대기 상태**: 백그라운드 작업 추적 시 SSE(`_waitForJobViaSse`)와 폴링(`_waitForJobByPolling`)을 혼용하고 있으나, 네트워크 불안정이나 백엔드 워커 메모리 초과 시 이벤트가 유실되면 UI 뱃지가 영구적으로 '검토 중' 상태에 머무르며 사용자의 추가적인 작업을 차단할 위험이 존재합니다.
* **오염된 컨텍스트의 재생산 통제 불가**: 정합성 판정 시 AI가 잘못된 환각(Hallucination)으로 거짓 모순을 지적할 때, 사용자가 무심코 '의도됨(whitelist)'으로 처리하면 이 정보가 강제로 팩트로 굳어집니다. 이를 되돌리는 별도의 관리 뷰가 없으므로, 장기적으로 RAG 인덱스가 오염되어 후속 에피소드의 정합성 검토 기능 전체가 마비될 수 있습니다.

* **정합성 검토 및 예외 처리 동작**: 정합성 모순 감지 시 '의도됨(승인)', '무시'를 선택할 수 있는 액션 팝오버(`action-popover`)가 노출된다.

**정합성 로직(`nf_consistency/engine.py`) 타당성 및 성능 평가**

* **설계 타당성**:
* 스키마 기반 사실 검증(명시 필드)과 NLI 모델을 통한 암시적 영역 검증(Layer 3)을 분리하여 3단으로 구성한 점은 요구사항을 충실히 반영한 형태다. `_judge_with_fact_index`를 통해 나이, 생사 여부 등 하드 제약을 우선 판별하고, 충돌 또는 불확실(UNKNOWN) 시 `_rerank_results_for_consistency`에서 NLI 스코어를 참조해 승급을 결정하는 논리적 흐름을 갖추고 있다.


* **성능 및 구조적 병목 (비판적 검토)**:
* **동기적 반복 쿼리로 인한 지연**: 800화 기준 정합성 검토 p95 지연시간이 7.65초로 목표(2.5초)에 미달한다. `_resolve_excluded_self_fact_eids` 함수 내에서 `chunk_size = 200` 단위로 루프를 돌며 DB 쿼리를 반복 실행하는 구조는 대규모 문서 스코프에서 I/O 병목을 유발한다.
* **비효율적인 Verification Loop**: 판정이 `UNKNOWN`일 경우 `verification_loop_max_rounds`만큼 FTS와 Vector 검색을 재시도하며 검색 반경(`k`)을 동적으로 늘려가는 구조(`_CANDIDATE_K + (round_idx + 1) * 2`)를 채택하고 있다. 이는 근거가 아예 존재하지 않는 '설정 구멍' 상황에서 최악의 지연시간을 강제하는 비관적 시나리오를 초래한다.


* **방법론의 한계점**:
* **단순 문자열 매칭의 의미론적 결함**: `_compare_slot` 내부의 텍스트 비교 로직은 `_STRING_EQUIVALENTS` 사전(예: "tomorrow": "nextday", "익일": "다음날")에 하드코딩된 값과 Jaccard 유사도(`_token_overlap_similarity`)에 극도로 의존하고 있다. 이는 문법적 변형이나 사전 외 동의어 처리에 취약하여 False Positive/Negative를 양산할 확률이 매우 높다.

**AI 방법론 도입 제안 및 비판적 검토**

현재의 단순 토큰 매칭 및 NLI 로직을 대체하고 정합성 성능을 높이기 위해 다음의 SOTA 방법론 도입을 검토할 수 있다.

1. **Late Interaction 기반 Reranking (ColBERT / Cross-Encoder)**
* **방법론**: 단순 코사인 유사도나 BM25 대신, 질의와 문서를 토큰 단위로 지연 교차 연산하는 ColBERTv2 아키텍처나, 텍스트 쌍을 직접 입력받아 모순/수반을 판단하는 Cross-Encoder(예: DeBERTa-v3 기반 NLI 모델)를 검색 후단의 Reranking 파이프라인으로 전면 채택한다. (관련 SOTA: *Khattab et al., "ColBERT: Efficient and Effective Passage Search via Late Interaction over BERT"*)
* **장점**: Jaccard 토큰 중복도나 하드코딩된 동의어 사전을 완전히 대체하며, 문맥적 의미와 서술 트릭이 포함된 복잡한 모순을 고정밀도로 식별할 수 있다.
* **비관적 단점 (최악의 상황)**: 연산량이 막대하다. 현재 p95 7.65초의 지연시간이 발생하는 로컬 환경의 제약 상, 매 Claim마다 Cross-Encoder 추론을 수행하면 시스템 전체가 OOM(Out of Memory)에 빠지거나 지연시간이 수십 초 단위로 폭증할 수 있다.


2. **Self-RAG를 변형한 Graph-augmented Entity Resolution**
* **방법론**: 현재 구현된 기초적인 Graph 확장을 넘어, 지식 그래프(Knowledge Graph)의 트리플(주어-동사-목적어)과 RAG를 결합하여 문서 검색 시점부터 엔티티 간의 관계 제약(Constraint)을 모델에 주입한다. (관련 연구: *Asai et al., "Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection"*)
* **장점**: 암시적 정보("왕궁으로 향했다" -> "수도에 있다")를 그래프 홉(Hop)을 통해 추론할 수 있어, `UNKNOWN` 판정 비율을 급격히 낮출 수 있다.
* **비관적 단점**: 그래프 인덱싱 과정 자체가 리소스를 과도하게 소모한다. 초기 인덱싱(Ingest) 시점에 모든 문서의 텍스트를 트리플로 추출하는 과정에서 오류가 섞이면(오염), 오히려 정합성 검증 엔진 전체가 '확신에 찬 환각'을 일으키며 전체 시스템의 신뢰도를 무너뜨리는 단일 실패점(Single Point of Failure)이 될 수 있다.



**해결 방향**: 성능 병목을 우선 피하기 위해, ColBERT와 같은 고비용 모델을 상시 가동하는 대신 현재의 `triage_anomaly_threshold` 스코어링을 경량화된 bi-encoder 로직으로 최적화하고, `UNKNOWN`이 확실시되는 고위험 Claim에 한해서만 사용자 동의(옵션) 하에 API를 통한 고성능 NLI 판정을 위임하는 방식(Fallback Routing)으로 설계를 제한해야 한다.
