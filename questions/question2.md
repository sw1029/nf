검토 기준은 user_request.md의 12개 항목(+ 8-1, 8-2 추가 요구)을 “현재 user_ui(웹 UI)에서 사용자가 실제로 할 수 있는 작업 흐름/피드백/오류 회복(undo)/가시성” 관점으로 대조하는 것입니다. 레포 내부에 이미 12항목 전수 판정 매트릭스가 있고(충족 6, 부분 8, 미충족 0), 아래는 그 판정을 UI/UX 관점에서 더 “사용자 입장에서 생길 수 있는 실패 모드(최악 상황)” 중심으로 재정리한 결과입니다.

현재 UI의 골격은 (1) 프로젝트 선택 모달, (2) 좌측 문서 내비게이션(본문/설정/구상/타임라인 탭 + 새 문서/새 챕터), (3) 중앙 에디터/툴바(백그라운드 작업/상시 점검 상태 배지 포함), (4) 우측 “도우미” 사이드바(점검/검색/제안)로 구성됩니다.

---

1. 설정/플롯/등장인물 문서 작성/분류 (요구 1) — “충족(기능은 됨), 운영 UX는 취약”
   요구 자체는 “설정·플롯·등장인물 문서를 별도로 작성/분류”이고, UI는 탭(본문/설정/구상/타임라인)과 새 문서/새 챕터 생성 버튼을 제공해 기본 분리는 됩니다. 또한 설정/구상 탭에서 placeholder를 바꿔 “어떤 내용을 쓰면 점검에 도움되는지”를 유도합니다.
   미비/리스크(UX):

* “문서가 많아질 때” 탐색 UX가 급격히 나빠질 가능성이 큽니다. 매트릭스도 대규모 문서 운용 UX를 리스크로 잡고(검색/필터 보강 필요), 현재 HTML/테스트 기준으로도 문서 트리 검색/필터 UI가 핵심으로 드러나지 않습니다(계약 테스트가 주로 컨트롤 존재/동작만 확인).
  최악 상황: 설정 문서가 수십~수백개로 늘면 “어느 설정이 근거로 쓰였는지/어디서 충돌났는지” 찾는 비용이 커져, 정합성 시스템이 있어도 사용자가 신뢰를 잃습니다(원인 추적 실패).

2. 작성 문서 인덱싱/임베딩(RAG) (요구 2) — “충족, 가시성은 부분”
   요구는 인덱싱/임베딩을 통한 RAG이고, UI는 저장 후 파이프라인(INGEST → INDEX_FTS)을 자동으로 트리거하고, 이후 백그라운드 정합성까지 이어집니다. 사용자는 “백그라운드 작업” 패널과 배지에서 작업 상태를 보게 되어 있습니다.
   미비/리스크(UX):

* “인덱싱이 지금 뭐하고 있는지”는 배지/패널로만 간접적으로 보입니다. 대규모 인덱싱이 걸릴 때(1000화급) user_request가 우려한 프리징 시나리오에서, 사용자에게 ‘진행률/병목/남은 작업량’이 충분히 설명되지 않으면 “멈춘 것처럼 보임”으로 이탈합니다.
* “재시도” UX가 사실상 모의(mock) 상태입니다. 실패 작업에 재시도 버튼을 보여주지만 실제로는 cancel을 때리고 “프론트엔드 모의” alert을 띄웁니다. 이 상태로 배포되면 신뢰 하락이 큽니다.

3. 정합성 위배 구간 segmentation + 신뢰성 + 예외처리 (요구 3) — “핵심은 구현, ‘설명 가능성/회복 가능성’이 부족”
   요구는 (a) 위배 구간을 segmentation해서 보여주고, (b) 신뢰성 점수를 근거 분해(FTS 강도/근거 수/확정근거/모델 점수)로 제공, (c) 동일 제안 반복 방지를 위해 문장 단위 예외처리(화이트리스트/ignore)까지 포함합니다.
   현재 UI/동작 근거:

* 문장 단위로 텍스트를 자르고(문장부호/개행 기반), 변경된 구간만 골라 큐에 넣고, 백그라운드 CONSISTENCY 잡을 돌려 결과를 갱신합니다.
* 결과는 verdict 카드로 렌더되고, 본문 하이라이트(CSS Highlights 또는 fallback span)로 “문제 구간을 눈으로” 연결합니다.
* 각 verdict 카드에서 “본문에서 보기/상세 근거/예외 처리/숨기기”가 제공되고, 예외처리(whitelist)와 무시(ignore)는 실제 API로 연결됩니다. 또한 클릭 위치에 따라 액션 팝업이 뜨는 “인라인 액션”도 구현되어 있습니다.
* 상세 근거 화면은 whitelisted/ignored 여부, claim fingerprint, unknown 사유, CONTRADICT 근거 수, fact_paths(태그 경로/엔티티)와 evidence(doc/tag/snippet)를 보여줍니다.
  미비/리스크(UX):
* “신뢰성 점수 분해”가 요구사항 핵심인데, UI 카드에는 현재 “신뢰도: overall”만 보이고, 분해 항목(FTS 강도/확정근거/모델 점수)이 사용자 언어로 설명되지 않습니다. 최악 상황은: 모델이 자신감 있게 틀린 판정을 내렸을 때(또는 evidence가 빈약할 때) 사용자는 숫자 하나를 근거로 오판하고, 시스템 전체를 불신하게 됩니다.
* “undo/되돌리기” UX가 사실상 없습니다. ignore/whitelist는 추가는 되지만, UI에서 목록 관리/해제(undo)가 보이지 않습니다(상세 근거에는 ignored/whitelisted boolean만 표시). 최악 상황: 사용자가 실수로 “숨기기”를 눌러 핵심 충돌을 영구히 못 보게 되어(또는 문제를 방치하게 되어) 장기적으로 설정 붕괴가 누적됩니다.
* segmentation 품질: 현재 문장 경계는 간단 규칙(문장부호/개행) 기반이라, 대화체/따옴표/서술 트릭이 많은 소설에서 “문장 분리 오류 → 엉뚱한 claim → 오탐/UNKNOWN 남발”이 쉽게 발생합니다. UNKNOWN이 많아질수록(매트릭스도 UNKNOWN 과다/설명 난해를 리스크로 지적) 사용자는 필터링 비용만 늘고 효용을 못 느낍니다.

4. 문법 교정(강도 조절) (요구 4) — “부분(제안 UI는 있으나 proofread UX는 빈약)”
   요구는 문법 교정을 강도 조절로 제공. 현재 도우미에는 제안(PROPOSE)에서 레벨에 따라 LOCAL_RULE/LOCAL_GEN을 나누는 흐름이 있고, strict 레벨 선택 시 경고 UI 토글도 있습니다.
   미비/리스크(UX):

* “문법 교정”이 사용자에게는 ‘정합성 점검’과 ‘문장 제안’ 사이에서 개념적으로 섞여 보일 가능성이 큽니다(실제로 버튼 라벨이 “현재 문단 다듬기 제안받기”로 표현됨).
* 강도별로 어떤 규칙/어떤 변형이 적용되는지(그리고 어디까지 자동 수정인지)가 UI에 명시되지 않으면, 사용자는 결과가 마음에 들지 않을 때 원인을 추적할 수 없습니다(매트릭스도 “레벨별 룰셋/평가 기준 명시”를 후속조치로 둠).

5. 자간/줄간격(레이아웃 설정) (요구 5) — “충족(UX 품질도 준수)”
   요구는 에디터 UX로 자간/줄간격 제공. UI는 config panel에서 글자 크기/줄 간격/자간/폰트를 슬라이더/셀렉트로 제공하고, CSS 변수 기반으로 에디터에 반영됩니다.
   리스크(UX):

* 브라우저별 폰트 렌더 차이/줄바꿈 차이로 페이지네이션 결과가 달라질 수 있습니다(매트릭스도 Edge 회귀를 언급). 최악 상황은 “페이지 경계가 달라져서 span 기반 verdict 하이라이트가 어긋나는” 케이스입니다.

6. 개선 제안 + 근거 인용(문서ID/섹션/태그경로) + 신뢰성 + 예외처리 (요구 6) — “부분(근거 표현이 요구 수준보다 약함)”
   요구는 “제안 시 근거를 문서ID/섹션/태그 경로로 보여줘야” 하고, 반복 방지도 필요합니다. 현재 PROPOSE는 SUGGEST job 결과의 citations를 “문서 제목 + snippet” 형태로 렌더합니다.
   미비/리스크(UX):

* user_request가 요구한 ‘태그 경로(예: 설정/인물/주인공/나이)’ 중심의 인용 표시가 UI에 기본값으로 노출되지 않습니다(현재 렌더는 title/snippet 위주). 최악 상황은 “사용자가 제안이 어디 설정을 근거로 했는지 확인 못해” 제안을 신뢰/채택하지 않게 되는 것입니다.
* “제안 결과”에 대한 예외처리(반복 억제) UX는 정합성 verdict에 비해 약합니다. ignore/whitelist는 verdict에 붙어있고, 제안에 대해서는 동일 수준의 suppress UX가 보이지 않습니다.

7. txt/docx export (요구 7) — “충족(기능은 확인), 품질 검증은 별도”
   요구는 txt/docx export. UI에 “내보내기” 버튼이 있고, 계약 테스트로 txt/docx만 노출됨이 확인됩니다.
   리스크(UX):

* docx는 환경별 폰트/줄바꿈 호환이 흔히 깨집니다(매트릭스도 샘플 교차 검증 권고). “내보낸 문서가 원고와 다르게 보이는” 순간 사용자 신뢰가 크게 떨어집니다.

8. n~m episode chunk 구성 + 기승전결/계량지표 (요구 8) — “부분(사용자가 ‘어디서’ 정의하는지 불명확)”
   요구는 n~m화 구간 지정 기반 episode chunk. 매트릭스도 “에피소드 기반 문서 운용/검색 필터 경로는 있으나 옵션 노출이 부족”으로 부분 판정입니다.
   미비/리스크(UX):

* 현재 UI는 “새 챕터/새 문서” 중심이라, 사용자가 “n~m화”를 언제/어디서 정의하고 그 정의가 검색/정합성/제안에 어떻게 적용되는지(필터, 범위, 우선순위)를 체감하기 어렵습니다.
* 최악 상황: 사용자는 episode chunk가 있다고 믿고 필터를 써도 결과가 기대와 달라 “그 기능이 없는 것처럼” 느끼고 이탈합니다.

8-1) time_key/entity_id 기준 grouping/filter (요구 8-1) — “부분(입력창은 있으나, 생성/승인 워크플로가 약함)”
요구는 time_key/entity_id 기준 그룹을 “사용자 요청 시에만 생성”하고, RAG/정합성/제안에서 필터링되게 하는 것입니다. UI는 entity/time/timeline 필터 입력과 quick/deep/strict 프리셋을 노출합니다(컨트롤 존재가 테스트로 확인). 또한 프리셋 값에 따라 graph_mode/triage/verifier/loop 파라미터가 자동 구성됩니다.
미비/리스크(UX):

* “그룹 생성(= time_anchors, entity_mentions 추출/링크)”은 user_request에서 ‘요청형’이 핵심인데, 현재 UI에는 ‘생성 실행/승인/재매핑’이라는 명시적 워크플로가 보이지 않습니다. 백엔드는 time-anchors/entity-mentions/timeline-events 엔드포인트까지 갖고 있지만, UI에서 이를 탐색/승인하는 흐름이 약하면 “필터 입력값을 사용자가 외워서 넣는 도구”가 됩니다.
* 최악 상황: 사용자는 entity_id/time_key를 모르는데 입력을 요구받고, 기능이 “전문가용 숨은 기능”이 되어 사실상 사용되지 않습니다(매트릭스도 ‘사용자 안내 부족’ 지적).

8-2) 세계관 타임라인 문서 지속 참조/확장 (요구 8-2) — “부분(뷰는 있으나 편집/매핑 UX 단절)”
요구는 세계관 타임라인 문서를 별도로 두고, time anchor가 timeline_idx를 참조할 수 있어야 합니다. UI는 “타임라인” 탭과 timeline-view 영역을 갖습니다.
미비/리스크(UX):

* 매트릭스가 말하듯 “타임라인 편집 워크플로 단절”이 가장 큰 문제입니다. 존재만 하는 타임라인은 ‘읽기 전용 덤프’가 되기 쉽고, 사용자가 실제로 매핑을 조정/승인하는 행위를 할 수 없으면 요구사항의 핵심(지속 확장)이 충족되지 않습니다.
* 최악 상황: 타임라인이 최신이 아니어서 정합성 체크가 과거 타임라인을 근거로 오판 → 사용자가 시스템을 끄게 됩니다.

9. 드래그 태깅 등 사용자 참여 기반 rule-base 병행 (요구 9) — “충족(메모/태깅 기반), 품질 진단 UX는 약함”
   요구는 사용자가 직접 태깅/참여하여 rule-base를 병행하는 것이고, 매트릭스는 “드래그 태깅/메모 UI 제공”으로 충족 판정입니다. 에디터는 ui_memos 메타데이터를 저장/로드하는 계약이 테스트로 고정돼 있습니다.
   미비/리스크(UX):

* user_request가 언급한 “태깅 품질 진단(희소/중복/일관성)”은 UI에 직접적인 피드백 루프가 보이지 않습니다(매트릭스도 후속조치로 남김). 최악 상황은 사용자가 태깅을 열심히 해도 “정합성 개선에 반영되는 느낌”을 못 받아 태깅 자체를 중단하는 것입니다.

10. 고용량 모델 선택적 사용 + API 키 (요구 10) — “충족, 보안 UX는 취약”
    요구는 고용량 모델을 선택적으로 쓰고, 아니라면 API 키를 받는 흐름. UI는 글로벌 설정에서 api key/model을 localStorage로 저장하고, 모든 API 호출에 Authorization 헤더로 붙입니다.
    미비/리스크(UX/보안):

* 키가 localStorage에 평문으로 저장됩니다. “로컬 앱”처럼 느끼는 사용자에게는 키가 안전하게 보관된다고 착각하게 만들 수 있고, XSS/브라우저 프로필 공유/백업 등에서 유출 위험이 있습니다. 최악 상황: 키 유출 → 과금/계정 손실 → 제품 신뢰 붕괴.

11. 정합성 모델 vs 생성 모델 분리 (요구 11) — “부분(모드 분리는 보이나, 정책 설명이 부족)”
    요구는 정합성(소형)과 생성/개선(고성능)을 분리. UI는 CHECK/PROPOSE 모드를 분리하고 버튼/패널이 바뀝니다.
    미비/리스크(UX):

* 사용자는 “왜 어떤 경우는 빠르고 어떤 경우는 느린지 / 왜 어떤 경우는 비용이 드는지”를 UI에서 이해하기 어렵습니다. 이 설명이 없으면, 최악 상황에서 사용자는 체크가 느리면 “성능 문제”로 오해하고, 제안이 틀리면 “정합성까지 못 믿겠다”로 확장됩니다(경계 정책 문서화 부족은 매트릭스 지적).

12. 정합성 3단 강제 근거화 + unknown 처리 (요구 12) — “부분(unknown은 보이나, 사용자가 ‘다음 행동’을 결정하기 어려움)”
    요구는 “근거화 3단 + 모르면 모른다”. UI는 UNKNOWN에 사유 텍스트를 표시하고, 프리셋(quick/deep/strict)에 따라 verifier/triage/loop까지 파라미터가 달라집니다.
    미비/리스크(UX):

* UNKNOWN이 “왜” UNKNOWN인지(근거 부족/근거 충돌/개체 식별 실패/시간 매핑 실패 등)와 “사용자가 무엇을 하면 UNKNOWN이 줄어드는지”가 UI에서 행동 단위로 연결되지 않습니다. 최악 상황: UNKNOWN이 쌓여서 사용자가 결과를 무시하고, 결국 정합성 시스템 자체가 꺼집니다(매트릭스도 UNKNOWN 과다/설명 난해를 리스크로 둠).

공통적으로 잘 된 점(UX 관점에서 “효용이 실제로 체감될 가능성”이 있는 부분)

* 백그라운드 점검의 “가시성”: 상단 배지(queued count, last time)와 상세 패널이 있어 ‘아무 것도 안 하는 것처럼 보이는 문제’를 줄입니다.
* “문제 구간 ↔ 본문” 연결: verdict 카드에서 span 기반 점프/하이라이트가 있고, CSS.highlights 기반 하이라이트(지원 안 되면 fallback)로 구현되어 브라우저 호환도 고려했습니다.
* 최근에 보고된 UI 회귀(도우미 닫기/X/Enter/팝업 좌표 등)는 스모크 테스트 기준으로 해결됐고, IME 등 일부는 수동 확인 필요로 남겨둔 상태입니다.

요약하면, user_request가 요구한 “작문 중 정합성 위배를 눈으로 확인하고(세그먼트+하이라이트), 근거를 보고, 필요하면 예외/무시로 반복을 끊는” 핵심 경험은 구현되어 있습니다. 다만 UI/UX 관점에서 치명적인 약점은 (1) 신뢰성 점수의 ‘분해 설명’ 부재, (2) ignore/whitelist의 되돌리기/관리 부재(회복 불능 UX), (3) time_key/entity_id/timeline_idx 같은 고급 기능이 “사용자 워크플로로 안내되지 않고 입력창/필터로만 남는” 구조, (4) 작업 패널의 재시도 기능이 실제가 아닌 점, (5) API 키 저장 방식의 보안 UX 취약입니다.
