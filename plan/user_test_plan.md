# User UI 테스트 계획 및 실행 절차

이 문서는 `user_ui.html`을 기반으로 한 기능, 데이터 무결성, 정합성, UI/UX 테스트를 위한 구체적인 계획입니다.

## 1. 테스트 목적
- User UI의 전반적인 기능(문서 생성, 편집, 저장) 동작 확인
- 대용량 데이터(소설 텍스트) 입력 시의 안정성 및 데이터 보존(Persistence) 검증
- 전역 정보(설정, 인물)와 본문 간의 정합성 위배 탐지 기능(Consistency Check) 검증
- UI/UX 설정(폰트, 반응성) 및 Edge Case(입력값 검증 등) 테스트

## 2. 테스트 환경
- **URL**: `http://127.0.0.1:8085/` (또는 로컬 서버 주소)
- **Browser**: Chrome (Headless 또는 GUI 모드)
- **Test Data**: `test_files/무한의 마법사 001-799 01-32.txt` 중 일부 발췌

## 3. 테스트 시나리오 (To-Do Checklist)

### Phase 1: 기본 기능 및 프로젝트 설정 (Project Setup)
- [ ] **TC-1.1**: 웹 UI 접속 (`http://127.0.0.1:8085/`) 및 로딩 확인
- [ ] **TC-1.2**: 새 프로젝트 생성 (이름: "Testing Regression")
- [ ] **TC-1.3**: 프로젝트 생성 후 메인 화면 진입 확인 (사이드바, 툴바 표시)

### Phase 2: 데이터 입력 및 보존 (Persistence Test)
- [ ] **TC-2.1**: '설정(SETTING)' 탭에서 새 문서 생성 (제목: "주인공 설정")
- [ ] **TC-2.2**: 설정 내용 입력 및 저장
    - 내용:
      ```
      이름: 이성민
      나이: 14세 (신체 나이), 정신 연령은 +13년
      직업: 노 클래스 (No Class)
      특이사항: 무공이나 마법 재능 없음. 13년 전으로 회귀함.
      ```
    - '저장됨' 상태 변경 확인
- [ ] **TC-2.3**: '본문(EPISODE)' 탭에서 새 문서 생성 (제목: "제1화 - 프롤로그")
- [ ] **TC-2.4**: 대용량 텍스트(`test_files/쥐뿔도 없는 회귀...txt`) 입력
    - 입력 텍스트 (발췌):
      ```
      생각해 보면, ‘시작’이라는 것은 참으로 불공평하고 부조리하기 짝이 없다.
      누군가는 태어나면서부터 손에 금수저를 쥐고 있다.
      ...
      [13년 전의 스타트 라인으로 돌아갑니다.]
      ```
    - 입력 후 스크롤링, 편집 반응성 확인
    - 자동 저장(Auto-save) 동작 확인 (5초 대기)
- [ ] **TC-2.5**: 브라우저 새로고침(Reload) 후 프로젝트 재진입
- [ ] **TC-2.6**: 저장된 '설정'과 '본문' 내용이 그대로 유지되는지 확인

### Phase 3: 정합성 및 오류 검증 (Consistency & Error Check)
- [ ] **TC-3.1**: 본문 내용 강제 수정하여 모순 발생시키기 (직업 모순)
    - 입력: "이성민은 태어날 때부터 9서클 마법사였으며, 천재적인 재능을 가지고 있었다."
    - 기대 결과: 설정의 "노 클래스", "재능 없음"과 충돌 감지.
- [ ] **TC-3.2**: 시점/나이 모순 발생시키기
    - 입력: "지금 이성민의 나이는 50세다."
    - 기대 결과: 설정의 "14세"와 충돌 감지.
- [ ] **TC-3.3**: 우측 '작가 도우미' 사이드바에서 '오류 점검' 탭 선택 -> '지금 점검하기' 클릭
- [ ] **TC-3.4**: 점검 결과(Analysis Result) 확인
    - "직업 설정 위배" 또는 "나이 설정 위배" 알림 확인.

### Phase 4: UI/UX 및 Edge Case
- [ ] **TC-4.1**: 화면 설정(Config Panel) 열기
    - 폰트 크기 변경, 줄 간격 변경 시 에디터에 즉시 반영되는지 확인
- [ ] **TC-4.2**: 제목 없는 문서 생성 시도 (Empty Title Handling)
    - 저장 시 제목이 없으면 자동으로 "무제" 혹은 경고가 뜨는지 확인.
- [ ] **TC-4.3**: 통합 테스트 (Organic Test)
    - 위 과정을 수행하며 에러 로그(콘솔) 확인.
    - 탭 전환 시 데이터 손실 없는지 재확인.

### Phase 5: 최종 보고
- [ ] 테스트 결과를 `verify/test_results.md`에 기록 (성공/실패 여부, 로그)

## 4. 실행 계획 (Execution Plan)
1. `browser_subagent`를 사용하여 위 시나리오를 순차적으로 수행한다.
2. 각 단계마다 스크린샷 또는 DOM 요소를 읽어 상태를 검증한다.
3. 실패 시, 어떤 단계에서 실패했는지 상세히 기록한다.
4. `run_debug_ui.py`가 실행 중이어야 하므로, 필요 시 재실행하거나 기존 프로세스에 접속한다.

## 5. 예상되는 결과 (Expected Outcome)
- 프로젝트 생성 및 문서 저장이 정상적으로 이루어져야 함.
- 새로고침 후에도 데이터가 보존되어야 함.
- 설정과 다른 내용을 본문에 입력했을 때, Assistant가 이를 감지하고 리포트해야 함.
- UI 설정 변경이 즉각적으로 반영되어야 함.
