# UI 디자인 분리 명세 (2026-02-27)

## 목적
- 본 문서는 `questions` 주장 검토 결과 중 `UI_DESIGN_SPEC` 태그 항목만 분리한 설계 명세이다.
- 구현/엔진 로직 변경은 제외하고, 사용자 인터랙션/레이아웃/가시성/회복성(undo) 명세만 다룬다.
- 원본 매핑: `plan/question_claim_mapping_2026-02-27.md`

## 공통 설계 원칙
1. 사용자 행동 수 최소화: 핵심 작업(점검 해석, 예외 관리, 범위 지정)을 2~3클릭 내 완료.
2. 컨텍스트 보존: 본문 포커스를 잃지 않고 세부 근거/예외 처리 수행.
3. 회복 가능성: 모든 위험 액션(무시/승인)은 되돌리기 가능해야 함.
4. 설명 가능성: 점수/사유/근거를 1차 카드에서 읽을 수 있어야 함.

## UI-01 신뢰도 분해 카드
- 관련 Claim IDs: `Q1-C06`, `Q2-C07`
- 현재 상태:
  - 백엔드는 breakdown(fts_strength, confirmed_evidence, model_score) 계산/저장.
  - UI 카드는 `reliability_overall` 단일 값 중심.
- UX 문제:
  - 사용자는 판정 근거의 구성 요소를 확인하기 어렵다.
- UI 명세:
  - verdict 카드 상단에 `Overall` + 분해 3항목을 막대/숫자 혼합으로 노출.
  - 항목 툴팁: 계산식 요약(정량 범위 0..1, 값 해석).
- 수용 기준(AC):
  - 카드 1개에서 `Overall/FTS/Confirmed/Model` 4개 값 확인 가능.
  - 상세 보기 진입 없이 핵심 판단 가능.

## UI-02 예외 관리 대시보드(whitelist/ignore/undo)
- 관련 Claim IDs: `Q1-C08`, `Q1-C12`, `Q2-C08`
- 현재 상태:
  - 개별 verdict에서 whitelist/ignore 추가만 가능.
  - 사용자용 목록/삭제/수정 UI가 없다.
- UX 문제:
  - 오판/실수 승인 복구가 어렵고 상태 추적이 불가.
- UI 명세:
  - 우측 사이드바 하위 탭 `예외 관리` 추가.
  - 목록 컬럼: claim_text, scope, kind, created_at, annotation(intent_type/reason/meta), 상태 토글.
  - 액션: `해제(DELETE)`, `수정`, `원문 점프`.
- 수용 기준(AC):
  - 추가된 예외를 목록에서 조회/검색/해제 가능.
  - 해제 후 verdict 목록이 즉시 재반영.

## UI-03 시점/인물/타임라인 매핑 워크플로
- 관련 Claim IDs: `Q1-C07`, `Q2-C17`, `Q3-C20`
- 현재 상태:
  - 필터 입력창은 있으나 생성/승인/재매핑 흐름 없음.
- UX 문제:
  - 사용자가 entity_id/time_key/timeline_idx를 기억해야 하며 발견성이 낮다.
- UI 명세:
  - 3단 Wizard: `추출 실행 -> 후보 검토 -> 승인/재매핑`.
  - 후보 목록에서 span 하이라이트와 함께 승인 버튼 제공.
  - 승인 결과를 필터 프리셋으로 저장.
- 수용 기준(AC):
  - 신규 사용자도 ID를 외우지 않고 필터 적용 가능.
  - 승인된 항목만 점검/검색 필터에 사용됨이 UI로 확인 가능.

## UI-04 n~m 범위 분석 지정
- 관련 Claim IDs: `Q1-C21`, `Q2-C16`, `Q3-C10`
- 현재 상태:
  - 문서 전체/변경구간 점검만 기본 제공.
- UX 문제:
  - 에피소드 범위를 직접 지정해 비교/점검하기 어렵다.
- UI 명세:
  - CHECK 패널에 `범위 모드(전체/현재 문서/n~m)` 추가.
  - n~m 입력 검증(정수, n<=m), 최근 사용 범위 저장.
- 수용 기준(AC):
  - n~m 입력으로 점검 job payload가 범위 기반으로 제출.
  - 범위가 결과 카드 제목/상태 배지에 표시.

## UI-05 jobs/consistency 상태 가시성 고도화
- 관련 Claim IDs: `Q2-C04`, `Q2-C05`
- 현재 상태:
  - 배지/목록은 있으나 재시도 동작이 mock이고 단계 설명이 약함.
- UX 문제:
  - 실패 대응 신뢰도가 낮고 진행 상태 해석이 어려움.
- UI 명세:
  - 상태 모델을 `QUEUED/RUNNING/SUCCEEDED/FAILED/CANCELED/PAUSED`별로 표준 뱃지화.
  - `재시도`는 실제 API 성공 시에만 활성화.
  - reason_code 노출 패널(메모리 압력/정책 제한/검증 실패).
- 수용 기준(AC):
  - 실패 작업 재시도가 실제 새 작업으로 전이되는 것이 UI에서 확인됨.
  - mock 알림 문구 제거.

## UI-06 제안(PROPOSE) 근거 카드 확장
- 관련 Claim IDs: `Q2-C13`, `Q2-C14`
- 현재 상태:
  - title/snippet 중심 렌더, 제안 suppress 제어 없음.
- UX 문제:
  - 제안의 근거 경로 추적성과 반복 억제 제어가 부족.
- UI 명세:
  - citation 카드에 `doc_id/section_path/tag_path/snippet` 표시.
  - 제안 항목별 `다시 보지 않기(suppress)`/복원 액션 추가.
- 수용 기준(AC):
  - 제안 카드에서 태그 경로를 기본 확인 가능.
  - suppress 처리 후 동일 제안이 즉시 숨김되고 복원 가능.

## UI-07 문법 교정(PROOFREAD) 전용 모드 분리
- 관련 Claim IDs: `Q2-C10`
- 현재 상태:
  - user_ui는 PROPOSE 중심이고 PROOFREAD 전용 흐름이 없다.
- UX 문제:
  - 문법 교정과 창작 제안의 목적 경계가 흐려진다.
- UI 명세:
  - 도우미 탭에 `교정(PROOFREAD)` 분리.
  - 레벨별 효과/자동수정 범위를 명시.
- 수용 기준(AC):
  - PROOFREAD 실행 시 SUGGEST와 별도 결과 카드/액션 사용.
  - 사용자가 모드 목적을 문구만으로 구분 가능.

## UI-08 API 키 보안 UX
- 관련 Claim IDs: `Q2-C20`
- 현재 상태:
  - localStorage 평문 저장.
- UX 문제:
  - 키 보관 위치/위험에 대한 사용자의 오해 가능.
- UI 명세:
  - 저장 정책 선택 UI: `세션만`, `OS 키링`, `저장 안 함`.
  - 첫 저장 시 위험 안내 모달.
- 수용 기준(AC):
  - 기본값은 세션 저장.
  - 영구 저장 선택 시 명시적 동의 절차 필요.

## UI-09 UNKNOWN 사유별 다음 행동 가이드
- 관련 Claim IDs: `Q2-C22`
- 현재 상태:
  - unknown reason 문자열은 보이지만 조치 유도가 없다.
- UX 문제:
  - 사용자가 다음 행동을 결정하기 어렵다.
- UI 명세:
  - reason별 CTA 제공:
    - `NO_EVIDENCE`: 근거 문서 추가/태그 지정
    - `AMBIGUOUS_ENTITY`: 엔티티 매핑 검토
    - `CONFLICTING_EVIDENCE`: 예외 관리 또는 사실 승인 검토
- 수용 기준(AC):
  - UNKNOWN 카드에서 1클릭으로 권장 작업 진입 가능.

## UI-10 타임라인 메타 편집(episode_no/time_key/timeline_idx)
- 관련 Claim IDs: `Q2-C18`, `Q3-C09`
- 현재 상태:
  - 문서 컨텍스트 메뉴는 `episode_no` 중심.
- UX 문제:
  - 타임라인 뷰를 유지/확장하는 편집 루프가 약함.
- UI 명세:
  - 문서 메타 편집 패널에 `episode_no`, `time_key`, `timeline_idx` 통합 제공.
  - 입력값 유효성 검사와 충돌 경고 표시.
- 수용 기준(AC):
  - 저장 즉시 타임라인 뷰 정렬/레이블이 반영.
  - 잘못된 인덱스 입력 시 저장 차단.

## UI-11 태깅 품질 진단 보조
- 관련 Claim IDs: `Q2-C19`, `Q3-C08`
- 현재 상태:
  - 인라인 태깅은 가능하나 품질/저장 연결 피드백 약함.
- UX 문제:
  - 태깅 노력이 정합성 개선으로 이어지는지 체감하기 어렵다.
- UI 명세:
  - 태그 진단 패널: 희소/중복/미연결(span only) 태그 경고.
  - 저장 성공 시 `schema 반영됨` 배지 표시.
- 수용 기준(AC):
  - 태그 상태를 `로컬 마크/저장됨/승인됨`으로 구분 표시.

## UI-12 모드 경계 정책 가시화(CHECK vs PROPOSE)
- 관련 Claim IDs: `Q2-C21`, `Q3-C11`
- 현재 상태:
  - 모드는 분리되어 있으나 실제 적용 범위/비용 설명이 제한적.
- UX 문제:
  - deep/strict 기대치와 실제 엔진 파라미터 간 괴리 발생.
- UI 명세:
  - 모드별 정책 카드 추가: 목적, 근거 범위, 예상 시간, 비용/품질 특성.
  - 레벨 선택 시 실제 파라미터 요약(읽기 전용) 표시.
- 수용 기준(AC):
  - strict 선택 시 적용 옵션 요약이 즉시 표시.
  - 사용자가 실행 전 기대 시간을 파악 가능.
